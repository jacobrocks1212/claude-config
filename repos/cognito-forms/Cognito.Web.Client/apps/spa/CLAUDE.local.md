# spa — Form Builder & Admin Interface

## Gotchas
- `GlobalState.ready` is a `Deferred<void>` — must await it before accessing state like `forms` or `organization`
- Composable naming (`composables/`): `use-*` prefix = reactive composable, `create-*` prefix = factory/async creator
- Element UI is a **forked** `@cognitoforms/element-ui` (dialogs, dropdowns, tooltips, form inputs) — import from it, not the standard `element-ui` package
- `GlobalState.forms` is `Record<string, UserFormMeta>` — metadata objects, not full Form entities
- Person-form auto-create action IDs are deterministic **per type** but NOT uniform across types (e.g. Guest's Submit can differ from Contact/User). Source action pickers from `Cognito.config.personFormTemplateActions[type]` (server-projected via `SystemTemplateService.GetTemplateActions`) — never hardcode action Ids or borrow from an arbitrary existing person form
- `CCallout` (`src/components/content/Callout.vue`) renders its root with `v-show="visible"`, so a bare `<CCallout>` is always in the DOM (just hidden). When a callout must be conditionally *absent* (e.g. a test asserting non-existence), gate it with `v-if` on your own condition, not the component's internal visibility. Use `type="error"` (validator-accepted) for the error variant; content goes in the default slot
- Callout/toast handlers (e.g. `showPersonFieldToast` in `identify-submitter.ts`) must **re-resolve live tokens at click time** inside the handler, never close over a snapshot from open time. A callout is opened once and refreshed in place via `updateCalloutOptions` (the handler is NOT re-registered), so a captured token list goes stale when fields are added later. Resolve the selected token from a fresh `getSingleSelectTokens()`; treat a provided-but-unmatched id as a no-op, not a default-to-first (bug 56579)
- Auto-create submitters: persisted `CreateNewPersonEntries` bool on `SubmitterPersonSettings` = the "Create New Person Entries" checkbox in `IdentifySubmitterSettings.vue`. **The server honors it** — `EntryIndexService.UpdateSubmitterPersonEntryByFieldMappingAsync` auto-creates only when `AutoCreateActionId.HasValue && CreateNewPersonEntries`. The checkbox defaults ON for viable forms (≥1 non-`WillAlwaysFail` action) while honoring an explicit OFF — hydration: `noViableActions ? !!CreateNewPersonEntries : (CreateNewPersonEntries !== false || AutoCreateActionId != null)` (brand-new vs explicitly-off viable forms both persist `false` — accepted residual gap). The Action picker gates on the checkbox **and** a non-empty action list (`showAutoCreateAction = showContactFieldDropdowns && internalCreateNewPersonEntries && autoCreateActionOptions.length > 0`). The save path can never persist the toggle ON without an action id (`handleIdentifySubmitterDialogSave` and the `IdentifySubmitterDialog` emit both gate on `actionId != null`). The zero-viable error callout is **deferred** — shown only when the toggle is on (`IdentifySubmitterSettings`) or `autoCreateBehavior === 'create'` (`IdentifySubmitterDialog`), never unconditionally
- The "Create New Person Form" inline flow seeds its action from the **new form's** viable set: `handlePersonFormCreated` → `selectDefaultActionValue(filterViableActionOptions(mapActionOptions(actions)))` → `handleIdentifySubmitterDialogSave`, setting `CreateNewPersonEntries` true iff a viable action exists. Changing the target form in Settings re-seeds the same pair — `SubmitterTargetFormChanged` derives the action from the new form's viable set (preserving a still-valid current id) and sets the flag iff an action resolved, atomically in the same `updateSettingsAndMarkDirty` round-trip. All four surfaces (Settings checkbox, Settings target-form change, Dialog create/skip, create-new-form) persist through the shared `identify-submitter.ts` wiring — do NOT re-implement filter/default logic; any new surface must seed both values together, never the flag alone. Unchecking the Settings checkbox also clears the action id (`createNewPersonEntriesChanged` emits `auto-create-action-change(null)` alongside `create-new-person-entries-change(false)`), so an explicit OFF persists as `{ CreateNewPersonEntries: false, AutoCreateActionId: null }` and stays off after the persist round-trip. The checkbox emit is bound in `Cognito.Services/Views/Shared/build.htm` → `Cognito.Forms.SubmitterCreateNewPersonEntriesChanged` (build.js) → `identify-submitter.ts`; runtime persistence of that checkbox is verified manually (no automated runtime coverage)
- `CExpandable` (`src/components/interaction/Expandable.vue`) skips its max-height animation when measured content height is 0 (e.g. expanding/collapsing under a `display:none` ancestor) — it settles state immediately instead of waiting for a `transitionend` that never fires. It still emits `'expanded'` (with `0`) on the skip path, so `@expanded` listeners must tolerate a zero height
- Stripe Connect embedded-component events (`setOnExit`, `setOnNotificationsChange`, …): capture `mountComponent`'s return and cast it to a local structural element type, since it's typed `Promise<unknown>`. **Gotcha:** with the injected `useStripeConnect()` variant you must `await` *before* casting (`await mountComponent(...) as SomeElement | null`). The `CognitoPayOnboardingEmbed.vue` `account-onboarding`/`setOnExit` precedent casts *without* `await` because it uses the `useStripeConnectProvider` variant — copying that await-less form for the injected variant silently gives you a Promise, not the element, so the handler never attaches
- **Vue 2.7 collection-in-`ref` is NOT reactive on mutation.** `ref(new Set())` (or `ref([])`) does not re-run templates/watchers on `.add()`/`.push()` — only `.value` *reassignment* is tracked. Replace the whole value: `const next = new Set(v.value); next.add(x); v.value = next`. Precedent: the `visitedTabs` lazy-mount latch in `EntryDetailsDockedTabbed.vue`. When latching off a required prop the parent round-trips via `.sync`, watch the **prop** (`watch(() => props.activeTab, …, { immediate: true })`), not the emit — the component re-emits `update:activeTab` and the parent feeds it back down

## State Management
`stores/global-store.ts` — single reactive `GlobalState` object (no Vuex/Pinia); async init signaled via `Deferred<T>` (Promise wrapper with external `resolve()`/`reject()`).

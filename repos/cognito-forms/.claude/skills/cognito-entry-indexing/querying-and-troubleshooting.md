# Querying and Troubleshooting

This document explains how the entry index is queried and provides guidance for troubleshooting common issues.

## Querying via the OData Endpoint

The primary way to query the entry index is through the `ODataController`. This controller exposes an OData (Open Data Protocol) endpoint that the frontend uses to populate the Entries page grid.

### The Query Flow

1.  **HTTP Request**: The frontend makes a GET request to an endpoint like `/api/odata/Forms(123)/Views(1)/Entries`. The request includes OData query parameters in the URL to specify filtering, sorting, and paging.
    -   `$filter=Status eq 'Complete' and Amount gt 100`
    -   `$orderby=DateSubmitted desc`
    -   `$top=50`
    -   `$skip=100`

2.  **Controller Action**: The `ODataController.GetEntryViewEntries` method receives the request.

3.  **State Creation**: The controller parses the OData parameters and uses them to construct an `EntrySetState` object. This object represents the complete state of the desired query (which form, which view, what filters, what sorting, etc.).

4.  **Service Call**: The controller calls `EntryIndexService.GetEntryIndexes(entrySetState, ...)`.

5.  **Repository Query**: The `EntryIndexService` passes the query information to the `IndexRepository`. The repository dynamically builds a SQL-like query string that is compatible with Azure Cosmos DB.

    *Example Generated Query (simplified):*
    ```sql
    SELECT c.id, c.i f, c.s['sort-id'] s, ...
    FROM c
    WHERE c.PartitionKey = 'org-id|form-id' and array_contains(c.i, 'filter-id')
    ```

6.  **Execution**: The query is executed against the Cosmos DB container, which efficiently returns the matching entry data.

7.  **Serialization**: The controller streams the results back to the client as a JSON response.

## Troubleshooting

### An entry is not appearing in search results or filters.

1.  **Check Index Status**: The most common cause is a stale index. If a form's structure was recently changed, the index for that form may have been invalidated and is awaiting a background rebuild. Check the `FormStatistics` entity for the form; if `IsValid` is `false`, a rebuild is pending.

2.  **Check Real-Time Indexing**: If the entry was just submitted, there could have been a failure during the real-time `UpdateLookups` process. Check application logs for any exceptions related to `EntryIndexService` or `IndexRepository` around the time of submission.

3.  **Examine the OData Request**: Use your browser's developer tools to inspect the full URL being requested by the Entries page. Ensure the `$filter` and other parameters are being constructed correctly based on your UI selections.

### A filter is not working as expected.

1.  **Verify Filter Expression**: The logic for a filter is defined in an `EntryView`'s `Filter` objects. The expression associated with that filter might be incorrect. Validate the filter's calculation logic.

2.  **Check the `IndexBuilder`**: The `IndexBuilder` class is responsible for evaluating the filter expressions and adding the corresponding `filter.Id` to the `Filters` array on the `CompositeEntryIndex` document. Debug the `FilterUpdate.WhenInvalid` method in `IndexBuilder.cs` to see if the filter expression is evaluating correctly for a specific entry.

### Search is slow or timing out.

1.  **Cosmos DB Performance**: The issue may lie with the performance of the underlying Cosmos DB instance. Check the Azure Portal for metrics on RU/s consumption, throttled requests, and query performance.

2.  **Query Complexity**: A very complex `$filter` can result in a less efficient query on the backend. Try simplifying the filters to see if performance improves. If it does, the query generated by `IndexRepository` may need to be optimized.

# Research: Safely extracting duplicated fail-open-guard scaffolding from a fleet of shell/Python hooks into a shared library — without turning the shared library into a single point of failure

## Research Question

We maintain a fleet of ~13 tool-invocation "hooks" (short shell scripts, seven of which embed an inline Python body invoked via `python3 -c "<body>"`). These hooks act as **security/safety guards**: they inspect a tool call's payload on stdin and either *allow* it or *deny* it by emitting a JSON `permissionDecision` object. The cardinal invariant is **fail-OPEN**: any internal error, missing dependency, or broken path must ALLOW the operation (never wedge the user's tooling) — a guard that fails *closed* is a worse bug than the gap it was guarding.

Roughly 20% of the hook plane (~467 of ~2,411 lines) is **duplicated scaffolding** copy-pasted across the hooks: a `python3`→`python` resolver, a script-directory derivation, JSON allow/deny emitters, an append-to-event-log function (with an inline fallback branch), a "breadcrumb" error-writer, and a pair of shared anchor regexes for matching command tokens. This copy-drift has already produced real, shipped bugs (a breadcrumb that wrote to the wrong variable and never worked; sibling hooks that never gained the error-trace tail their peers have; a matcher-semantics fix that had to be hand-landed in three files that each believed they owned the regex).

We want to extract this scaffolding into a shared **sourced bash prelude** (`hook-prelude.sh`) plus an **importable Python module** (`hook_lib.py`), then migrate the hooks one at a time, proving byte-identical allow/deny behavior after each migration with an existing 157-test pipe-test suite. The central tension: **introducing a shared dependency into a fleet of fail-open guards risks converting N independent, locally-degrading failure points into ONE correlated failure point** — if the shared library breaks, every guard breaks together. We need this research to pressure-test our approach against established practice, surface pitfalls we haven't anticipated, and recommend the safest concrete patterns.

## Context

- **Environment:** Git Bash on native Windows (not WSL), plus cross-platform (cloud Linux containers). Shell scripts must be POSIX-`sh`-portable; Python is stdlib-only, invoked as `python3 -c "<body>"` (never a heredoc — stdin carries the tool payload, so the Python body cannot read stdin for its own imports).
- **Current failure isolation:** today each hook is fully self-contained. A corruption or bug in one hook's scaffolding copy affects only that hook. The other twelve keep working. This *accidental* isolation is the property a shared library puts at risk.
- **Fail-open mechanics:** hooks source-guard with `. "<dir>/hook-prelude.sh" 2>/dev/null || exit 0` (a missing/broken prelude must `exit 0` = allow), and the Python body wraps `import hook_lib` so that an ImportError still degrades to a bare allow *and* leaves an error trace.
- **The "leave a trace" contract:** even on the fail-open path, the guard must best-effort append one line to an event log / write an error breadcrumb, so silent failures are observable. One of the bugs we're fixing is precisely a fail-open path that left *no* trace.
- **Latency sensitivity:** these run on *every* matching tool call, inside a ~5-second hook timeout budget. A separate measured concern (out of scope for this feature but relevant to the module's design) is that importing a large sibling module (`lazy_core`) per invocation costs ~95ms warm / up to ~670ms cold. The shared `hook_lib.py` must therefore stay **import-light** — stdlib-only at module top, heavy imports deferred lazily inside the functions that need them.
- **Behavior-change proof:** we have a 157-test suite that pipe-tests each hook's stdin→stdout (deny/allow JSON) byte-for-byte. "No behavior change" is operationalized as "all 157 still pass, output byte-identical," run after each single-hook migration.

## Baseline Approach Summary (what we currently plan to do)

1. **`hook-prelude.sh`** — a *sourced* (never executed) bash file providing: `python3`→`python` resolution into a variable, script-dir derivation, and a **pure-bash** no-Python fallback event/breadcrumb writer (printf/date only, so the trace survives even when Python is entirely unavailable). Sourced under a fail-open guard so a missing prelude still allows.
2. **`hook_lib.py`** — an importable Python module (sits beside the existing `lazy_core.py` so existing path-threading resolves it) providing: `allow()` / `deny(reason)` JSON emitters, `append_hook_event(...)` (delegating to the heavy `lazy_core` appender when importable, inline stdlib fallback otherwise), `breadcrumb(hook, err)`, and the shared anchor-regex constants — the single source for the matcher pair that is currently triplicated.
3. **One inline fallback retained per hook** — if `import hook_lib` fails, the hook still fails open correctly (bare allow) *and* leaves a trace via the bash-side prelude writer. Only the *rich* 40-line inline fallback collapses into the library; a minimal import-failed guard survives per hook.
4. **Migrate hook-by-hook**, lowest-risk first, full 157-test suite after each, byte-identical output asserted by the suite.

## Research Areas

Investigate each, grounded in established software-engineering practice, real-world precedents, and the specifics above:

1. **Shared-library-as-single-point-of-failure in fail-open guard fleets.** How do mature systems that run many independent safety/permission checks (e.g. PAM modules, sudo/`sudoers` plugins, git hook frameworks like `pre-commit`, OPA/Rego policy bundles, Kubernetes admission webhooks, browser-extension content-script shims, editor/LSP plugin hosts) share common scaffolding WITHOUT creating a correlated failure that takes down every check at once? What isolation patterns (vendoring per-consumer, versioned copies, defensive import guards, capability probing) are used, and what are their tradeoffs against the deduplication benefit?

2. **Fail-open vs. fail-closed for shared code paths.** Our per-hook code fails open by design. When common logic moves into a shared library, which failure modes should the *library* itself adopt at each layer (module import, individual function call, dependency-of-dependency import)? Where is fail-open correct and where does it hide bugs that should surface? What does the security literature say about the risk of a fail-open guard silently becoming a no-op (the "guard that always allows" degradation) and how to detect it in production?

3. **Refactoring N copies → 1 with a behavior-preservation guarantee.** What is the established discipline for this exact move — Michael Feathers-style *characterization/golden-master tests*, "make the change easy, then make the easy change," strangler-fig incremental migration, parallel-run/shadow verification? Given we already have a 157-test byte-identical pipe-test suite, what additional safeguards (mutation testing of the suite itself, differential testing old-copy-vs-library, property-based fuzzing of the matcher regexes) would meaningfully raise confidence that "byte-identical output" truly means "no behavior change"? What are the known blind spots of golden-master suites (they pin current behavior including current *bugs*)?

4. **The bash-prelude sourcing pattern.** Best practices and pitfalls for a *sourced* (not executed) shared shell library that must be POSIX-portable across Git-Bash-on-Windows and Linux: safe `SCRIPT_DIR`/self-path derivation for a *sourced* file (where `$0` is the caller, not the source), guarding against double-sourcing, namespace pollution / function-name collisions in the caller's shell, `set -e`/`set -u` interactions, and keeping the source-guard itself fail-open (`|| exit 0`). What are the portability traps specifically for Git Bash on Windows (path separators, `command -v` behavior, `date`/`printf` differences)?

5. **Import-light Python module design.** Techniques and idioms for keeping a frequently-imported stdlib-only module cheap: deferring heavy imports to call sites (lazy/function-local imports, `importlib` on demand, `__getattr__`-based lazy submodule loading, PEP 562), avoiding import-time side effects, and *testing* import-lightness (e.g. asserting a heavy dependency is NOT in `sys.modules` after import). What is the real per-invocation cost profile of `python3 -c` cold starts, and which of it is unavoidable interpreter startup vs. import cost we control?

6. **Making the "always leave a trace" contract robust.** Patterns for best-effort observability on a failure path that itself must not throw: append-only structured event logs (JSONL), atomic/lock-free appends from concurrent short-lived processes, degrade-gracefully-when-the-log-is-unwritable, and *pure-shell* logging when the richer (Python) logger is exactly what failed. How do other systems guarantee that a guard's fail-open path is *observable* rather than silent?

7. **Duplication vs. abstraction — when NOT to extract.** The countervailing view: "a little copying is better than a little dependency" (Go proverb), the AHA / "avoid hasty abstractions" principle, and the wrong-abstraction cost (Sandi Metz: "duplication is far cheaper than the wrong abstraction"). Given our copy-drift has a *demonstrated bug trail*, what criteria distinguish this as a case where extraction is clearly warranted vs. cases where the shared abstraction becomes a liability? Are there hybrid options (generate-and-vendor the scaffolding from a single source, lint-enforced identical copies, a code-generation step) that capture the "one source of truth" benefit while preserving per-consumer failure isolation?

## Specific Questions

1. For a fleet of independent fail-open safety guards, is a **shared runtime library** or a **single-source-generates-vendored-copies** model (each hook keeps its own copy, but all copies are mechanically generated/lint-verified identical) the better tradeoff between deduplication and failure isolation? Under what conditions does each win?
2. What concrete defensive patterns should wrap the `import hook_lib` boundary so that a corrupted/partial/missing library provably degrades to a correct fail-open allow *with a trace*, and never to a wedge or a silent no-op?
3. For the *sourced* `hook-prelude.sh`, what is the most robust portable idiom to derive its own directory when `$0` refers to the calling script, across Git Bash on Windows and Linux? What are the specific failure modes to test?
4. How can we test that `hook_lib.py` stays import-light over time (a regression guard), and what is a realistic budget given `python3 -c` interpreter-startup overhead that no design can remove?
5. Beyond a byte-identical golden-master suite, what verification would most raise confidence that a migration introduced *zero* behavior change — differential testing against the pre-migration copy? mutation testing to prove the suite actually constrains the scaffolding? fuzzing the shared matcher regexes against both implementations?
6. What ordering and rollback strategy minimizes blast radius when migrating guards one at a time — and what "circuit breaker" or fast-revert signals should we watch after each step?
7. Are there known incidents / postmortems (open-source or published) where consolidating duplicated guard/hook/policy scaffolding into a shared library caused a *correlated* outage or a fail-open-turned-fail-closed regression? What were the root causes and the recommended preventions?
8. For the shared anchor regexes (currently triplicated across three matchers), what is the safest way to establish a single source of truth while guaranteeing the three consumers' matching semantics remain byte-identical through the change — and how should the regexes themselves be tested (property-based? corpus of known allow/deny command strings)?
9. Is there value in a **version/health stamp** on the shared library (so a hook can detect it loaded an unexpected/old prelude and choose to fail open loudly), or does that add more fragility than it removes for a same-repo, same-commit shared file?
10. What lightweight, always-on **runtime canary** would detect the worst silent failure mode — a guard that has degraded into "always allow" — without adding meaningful latency to the hot path?

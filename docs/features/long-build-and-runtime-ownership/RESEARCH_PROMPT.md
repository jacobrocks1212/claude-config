# Owning a long-lived child process across an ephemeral agent-turn boundary (stdlib-only Python harness, Windows + WSL)

## Research Question

In an autonomous LLM-agent development harness, a "cycle subagent" runs for one turn and is then **reaped** — the orchestrator kills the subagent's entire process tree when its turn ends. Any long-running process a subagent starts (a packaged build, or a long-lived dev/HTTP server) dies with it. We need the canonical, battle-tested patterns for making a long-lived child process **provably owned by a controller that outlives the ephemeral worker that requested it**, so the process survives the worker's turn boundary, its ownership is *verifiable* later (not just "something answers on the port"), and a single deterministic primitive can assert/recover liveness instead of hand-rolled poll loops.

The harness is a **stdlib-only Python** control layer that shells out to OS primitives. It runs on **Windows 10/11 with Developer Mode on** and on **WSL/POSIX**. We are deliberately NOT pre-deciding the ownership mechanism — we want prior art and tradeoffs to pick between (a) controller-process + tracked handle/sentinel, (b) a detached/daemonized supervisor process, and (c) an OS-service-level runtime, across both platforms.

## Context

- **Tech stack of the controlled processes:** a packaged desktop-app build (`tauri build` / `cargo build --release`) that *terminates and produces an artifact*; and a persistent dev runtime (`tauri dev`) plus an MCP HTTP server bound to TCP 3333 that is *long-lived and health-polled* (`GET /health` → 200, plus an assert that a named MCP tool is present).
- **The controller layer:** a Python CLI (`lazy-state.py` → `lazy_core.py`) invoked by the orchestrator. It already has a working `ensure_runtime(repo_root, *, config, probe, restart, stale_check)` subcommand: probe `/health` → staleness check (native source newer than the boot stamp) → background `dev:restart` → bounded curl-until-200 → assert MCP tool present → return `{status, mcp_tools_present, health_code}`. It is **hermetic under test** via injected `probe`/`restart`/`stale_check` callables. We want any new mechanism to follow the same shape: shared impl in `lazy_core`, repo specifics in a parameterized config dict, stdlib-only, injectable callables for hermetic tests.
- **The failure modes observed (this is the problem we are solving):**
  1. **Reaped runtime** — the runtime was booted *inside a subagent's own subprocess*, whose process tree was torn down on return. A later cycle's health check meets a dead runtime. The tell was a `health_code: 0` (connection refused) even though "ensure-runtime" had reported success in the earlier cycle.
  2. **Orphaned production edits** — a long build backgrounded inside a subagent died with that subagent's turn, leaving real source edits uncommitted and the cycle orphaned.
  3. **Hand-rolled poll loops** — the orchestrator compensates by hand-composing rebuild → health-poll → inspect-telemetry until-loops, the same shape repeated 6+ times in a single session.
- **Existing enforcement seam:** a `PreToolUse` hook (`lazy-cycle-containment`) already denies a subagent a defined set of "orchestrator-only" operations (recursive dispatch, killing/restarting the dev server). Enforcement is **fail-OPEN** (a query error falls back to permitting). The subagent/orchestrator boundary is also bracketed by `--cycle-begin`/`--cycle-end` calls that record process-friction (torn bracket, unexpected commits) to a deny-ledger.

## Baseline Design (what we will build once research locks the mechanism)

1. **Make the ownership boundary enforceable, not advisory.** Today an advisory prose rule says "the orchestrator owns long builds; run them backgrounded from the main session, never from inside a cycle subagent." A subagent can and does violate it with no enforcement. The long-build path needs to be closed the same deterministic way the dev-server kill/restart already is.
2. **Make runtime ownership survive the boundary by construction and be verifiable.** Record the runtime's *owner* (a handle/sentinel keyed to the live controller session) so a later cycle can assert "this runtime is owned by the live orchestrator," not merely "something answers `/health`."
3. **One liveness/recovery primitive replaces hand-rolled loops.** A single deterministic call returns a structured verdict and, on un-recoverable failure, a blocker signal — the orchestrator stops hand-composing the until-loop.
4. **No orphaned production edits on a mid-build tear** — either the build survives the tear (prevent-by-ownership) or the orphan is detected and reconciled/surfaced (detect-and-recover), or both.

## Research Areas

- **Cross-platform process supervision for surviving an ephemeral controller.** The canonical OS primitives and their tradeoffs:
  - **Windows:** Job Objects (and `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` vs. *not* setting it / `CREATE_BREAKAWAY_FROM_JOB`), `DETACHED_PROCESS` / `CREATE_NEW_PROCESS_GROUP` / `CREATE_NO_WINDOW`, process-group semantics, and how a child outlives the spawning process vs. gets reaped with its parent's job. What is reachable from **stdlib-only Python** (`subprocess` `creationflags`, `os`, `ctypes` against `kernel32` if unavoidable) without third-party deps like `pywin32`?
  - **POSIX/WSL:** `setsid` / new session leadership, `nohup`, double-fork daemonization, process groups, `prctl(PR_SET_PDEATHSIG)` (and why it does NOT help when the *controller* is the one that must die), and supervisor equivalents (a `systemd --user` transient unit via `systemd-run`, `tmux`/`screen` detached session, `daemontools`/`runit`/`supervisord`). Which are stdlib-reachable vs. require an external binary?
  - WSL-specific gotchas: does a process detached under WSL survive the WSL controller's turn? Interop between Windows-spawned and WSL-spawned processes when the harness straddles both.
- **Prior art in CI systems and ephemeral-agent harnesses** for owning two process shapes — a *terminating, artifact-producing build* and a *long-lived, health-polled service* — across worker/controller lifecycles. How do GitHub Actions / GitLab runners / Buildkite agents, devcontainers, and testcontainers-style libraries manage service containers and background build steps so they outlive (or are deterministically torn down with) an ephemeral step? Do they use one mechanism for both shapes or two?
- **Verifiable ownership, not just liveness.** Patterns for recording *who owns* a process so a later actor can distinguish "owned by the live controller" from "orphaned but still answering" — PID + start-time fingerprinting (PID-reuse safety), a lock/handle file keyed to a session id, a port-plus-pidfile convention, advisory file locks (`flock` / `LockFileEx`), or a control socket. How do mature daemons (databases, language servers, dev servers) do pidfile + liveness so a stale or hijacked process is detected rather than trusted?
- **Structured liveness/recovery verdict + recovery contract.** What belongs in a liveness verdict beyond `{status, health_code, tool_present}` — ownership identity, staleness, restart-attempt outcome, a terminal "un-recoverable → blocker" signal? What should "recover" mean for: a *stale* runtime (source newer than boot — rebuild/restart), a *dead* runtime (gone — reboot), a *hijacked* runtime (something else on the port), vs. a *genuinely blocked* runtime (can't be brought up — surface a blocker, do not loop)? Idempotent / bounded-retry conventions for such a primitive.
- **Torn-build orphan handling: prevent vs. detect-and-recover.** When a worker is torn down mid-build: prevent-by-ownership (the controller owns the build so it survives the tear), detect-and-recover (the orphan/uncommitted delta is detected post-tear and reconciled or surfaced), or both. What do CI systems and build tools (incremental compilers, content-addressed build caches) do to make a torn build safe — atomic artifact promotion, lockfiles on the build dir, detecting a half-written output? How does this compose with a "no silent uncommitted source delta" guarantee in a git working tree?
- **One mechanism or two.** Whether a single ownership/supervision mechanism cleanly covers BOTH a terminating artifact build AND a persistent health-polled service, or whether prior art consistently separates them (e.g., build = job-object-tracked transient; service = supervised daemon). The tradeoffs in complexity, cross-platform behavior, and recoverability.
- **Fail-open enforcement of an ownership boundary.** Patterns for a *fail-open* guard that prevents/redirects an ephemeral worker from backgrounding a turn-crossing process — how to detect "this background-spawn would be reaped" at request time, and redirect to the controller, without false-positives that block legitimate short-lived subprocesses. Prior art in sandboxing / process-guard hooks.

## Specific Questions

1. **Central fork — which ownership level?** For a stdlib-only Python controller on Windows-Dev-Mode + WSL, which is the cleanest mechanism to make a child process provably outlive an ephemeral worker: (a) controller-process + a tracked handle/sentinel (PID+start-time+session-id pidfile), (b) a detached/daemonized supervisor the controller spawns and tracks, or (c) an OS-service-level runtime (Windows service / `systemd --user` unit)? Rank by complexity, cross-platform uniformity, recoverability, and how cleanly each integrates with an existing PID-less `ensure_runtime` + on-disk run-marker.
2. **Windows, no third-party deps.** From `subprocess.Popen` `creationflags` and `ctypes`/`kernel32` alone (no `pywin32`), what is the minimal reliable recipe to spawn a child that is NOT killed when the spawning Python process exits, AND that we can later positively identify as ours? Does `DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` suffice, or is an explicit Job Object with kill-on-close *cleared* / a breakaway required? What are the PID-reuse and orphan-cleanup hazards?
3. **POSIX/WSL, stdlib-reachable.** Is `os.setsid()` in a double-fork (or `start_new_session=True` in `subprocess`) sufficient for survival across the controller's death, or is an external supervisor (`systemd-run --user`, detached `tmux`) materially more robust for a multi-hour dev runtime? Any WSL-specific reaping behavior that breaks the naive `setsid` approach?
4. **Verifiable-ownership record.** What is the most robust stdlib on-disk ownership record — PID + process-start-time + a controller session id in a JSON sidecar, validated against `/proc` (POSIX) and a `ctypes` `GetProcessTimes` (Windows) — to defeat PID reuse and distinguish "my live runtime" from "a stale or foreign process on the port"? How do production daemons template this?
5. **Liveness verdict schema + recovery semantics.** What fields should a single deterministic liveness/recovery primitive return, and what is a clean state machine for `ready | booted | stale-rebuilt | hijacked | unrecoverable→blocked`? Where is the boundary between "auto-recover" (rebuild/restart) and "surface a blocker, stop looping"?
6. **Torn-build safety.** What concrete techniques make a build torn mid-flight safe (atomic artifact promotion, build-dir lockfiles, detecting partial output) and how do they combine with a guarantee of "no silent uncommitted source delta" in a git working tree — prevent-by-ownership vs. detect-and-recover, and the minimal addition on top of an existing torn-bracket friction detector?
7. **One mechanism or two — what does prior art do?** In CI / ephemeral-agent / devcontainer systems, is a single supervision mechanism used for both transient builds and persistent services, or are they consistently separated? What is the recommendation for our two specific shapes (`tauri build` artifact vs. `tauri dev` + MCP/3333 service)?
8. **Fail-open ownership-boundary guard.** How can a request-time guard reliably distinguish "this is a turn-crossing build/runtime spawn that must be redirected to the controller" from "this is a legitimate short-lived subprocess," with low false-positive rate, in a fail-open hook? Any prior art for classifying a spawn as long-lived/turn-crossing at request time?

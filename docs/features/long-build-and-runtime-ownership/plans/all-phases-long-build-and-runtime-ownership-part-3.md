### WU-1 — `long-build-ownership-guard.sh` PreToolUse hook + `settings.json` registration

**Scope (PHASES deliverables):** "`user/hooks/long-build-ownership-guard.sh` (net-new) — fail-OPEN PreToolUse Bash hook; inline Python matcher anchored to exact long-build signatures … on a match, emits a `deny` with a corrective message naming the orchestrator-takeover signature; ANY internal error → allow." + "Register the hook in `user/settings.json` as an additional entry in the existing `\"matcher\": \"Bash\"` PreToolUse array."

**TDD:** yes (hook-pipe tests: synthetic PreToolUse JSON on stdin → assert emitted deny/allow).

**Files to create/modify:**
- `user/hooks/long-build-ownership-guard.sh` — NET-NEW. Model the fail-open inline-Python shape on `user/hooks/block-noncanonical-blocker-write.sh` [VERIFY: `ls user/hooks/block-noncanonical-blocker-write.sh`] (read it: `-c`-passed inline Python, deny via `hookSpecificOutput`/`permissionDecision: deny`, ANY error → exit 0 allow + breadcrumb). NOTE: SPEC §M5 prose says "fail-open blocks (`exit 2`)"; in Claude Code a PreToolUse blocks via the JSON `permissionDecision: deny` (a non-zero exit is a hard error). Implement the BLOCK as a `deny` JSON whose reason names the orchestrator-takeover signature — that IS the "fail-open block" in this hook framework, matching the sibling guards. The matcher anchors: `^\s*(?:[A-Za-z_][A-Za-z0-9_]*=\S+\s+)*` prefix tolerance then one of `tauri build`, `cargo build --release`, `npm run build` (exact long-build binary invocations only — never `ls`/`cat`/short `npm run lint`).
- `user/settings.json` — add the hook command as a sibling entry in the `PreToolUse` `"matcher": "Bash"` hooks array [VERIFY: `grep -n '"matcher": "Bash"' user/settings.json`] (alongside `block-work-repo-git-push.sh`, `block-terminal-kill.sh`, `lazy-cycle-containment.sh`).
- A hook-pipe test file under `user/hooks/` or `user/scripts/` (follow the existing hook-test convention — grep for how other `.sh` hooks are tested, e.g. a `test_*hook*` or a pipe fixture).

**Test files:** the hook-pipe test (synthetic-payload-on-stdin harness; match the existing guard-test style).

**Test expectations:**
- `command: "tauri build"` → deny JSON present, reason names the takeover signature.
- `command: "cargo build --release"` → deny.
- `command: "npm run build"` → deny.
- `command: "ENV=1 tauri build"` (leading env assignment) → deny (prefix tolerance).
- `command: "ls -la"`, `command: "cat foo"`, `command: "npm run lint"`, `command: "cargo check --release"` → allow (no deny output) — false-positive scope guard.
- Malformed payload (bad JSON on stdin) → fail-open allow (exit 0, no deny), with a breadcrumb (mirror the sibling guards).

**Implementation goal:** The M5-Prevent request-time guard — redirect exactly the long-build signatures to orchestrator ownership so the build survives a subagent tear by construction, with a tight false-positive scope. Fail-OPEN like every sibling guard.

**Spec requirements:** SPEC M5 Prevent; LD4; "scope the matcher to exact long-build binary invocations".

**Batch:** 1.

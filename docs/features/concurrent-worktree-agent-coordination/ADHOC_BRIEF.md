# Ad-hoc task: Concurrent multi-agent worktree coordination (awareness + FIFO file-lock + conflict handling)

Make concurrent multi-session agent work on a SHARED worktree/branch safe and non-panicking, in BOTH claude-config and AlgoBooth. Motivated by a live 2026-07-18 /lazy-batch run where the operator's concurrent session committed 28 archives to main mid-cycle, tripping a false process-friction and nearly triggering a false breach alarm. SIX requirements (each a design surface for /spec to walk with the operator): (1) AWARENESS — every agent (cycle subagents, orchestrators, and their sub-subagents) in claude-config + AlgoBooth must be told, in their dispatch/context, that OTHER agents may be working the same worktree/branch concurrently from parallel sessions; bake this into the cycle-base-prompt + relevant skill contracts + CLAUDE.md. (2) GIT SAFETY — make all git operations as safe as possible under an open shared worktree: fetch+ff-before-push everywhere, bounded push-retry on non-ff, never force, prefer append-only / pathspec-scoped commits (the flush-commit-artifacts precedent), avoid blanket 'git add -A' that absorbs a concurrent writer's staged files. (3) FIFO FILE-LOCK — agents DETECT write conflicts and coordinate via a FIFO/queue locking system so each agent proceeds in turn, waiting on a file to unlock before proceeding; borrow/share code with the build-queue system (build-queue.ps1 FIFO serializer, and/or lazy_coord.py's os.mkdir global lock + fencing-token leases + owner.json stale-holder reclamation). Cross-platform (build-queue is PowerShell/workstation; need a stdlib-Python path for cloud/AlgoBooth too — mirror the runner-outcome-contract two-implementation pattern). (4) NON-HALTING WRITE CONFLICTS — a write conflict must NOT halt a lazy-batch run; treat as rare-but-expected in the long run, handled consistently and without panic (retry/queue, log, continue). (5) SEMANTIC CONFLICTS → HALT — when a genuine SEMANTIC conflict arises (the two agents' work is logically incompatible so work should pause), write NEEDS_INPUT.md (class product) and HALT; NO provisional auto-accept even under --park-provisional (add to the lazy_core.provisional_eligibility fail-closed carve-out, like the Spike-FAIL precedent). (6) LARGE/COMPLEX NON-SEMANTIC CONFLICTS → TEMP-WORKTREE MERGE-BACK — when a large/complex but NON-semantic conflict arises, the orchestrator creates a temporary worktree to complete the work, then merges back and resolves conflicts as needed; if this agent BEATS the conflicting agent to the merge, it must COMMUNICATE to the conflicting agent (e.g. via a commit message the other agent is bound to see on its next fetch/rebase, or a sentinel/marker the other agent reads) that conflicts should be expected + how to resolve them; this path must NOT halt a lazy-batch run. Cross-repo: claude-config (the harness) is the primary home; AlgoBooth consumes the same contracts. Design forks to surface at /spec: the lock granularity (per-file vs per-dir vs per-item), the semantic-vs-non-semantic conflict discriminator, the temp-worktree lifecycle + merge-back protocol, and the cross-agent communication channel. Origin: /lazy-batch run 2026-07-18, operator directive.

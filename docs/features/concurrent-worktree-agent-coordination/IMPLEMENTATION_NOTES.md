## Phase 2 — Git safety (fetch+ff-before-push, bounded non-ff retry, pathspec commits, friction carve-out)

#### Implementation Notes (Phase 2 — Batch 1: WU-1 + WU-2)
**Completed:** 2026-07-18 (Batch 1; WU-3 prose is Batch 2)
**Work completed (Batch 1, file-disjoint parallel dispatch — 2 test + 2 impl Sonnet sub-subagents):**
- **WU-1** `git_safe_push(repo_root, *, branch=None, remote="origin", run=None, sleep=None, max_retries=3) -> dict` added to `lazy_core/runtimeplane.py` (@2488), registered in the PEP-562 facade map `lazy_core/__init__.py` (@560, `"git_safe_push": "runtimeplane"`). Each attempt: `fetch <remote>` → `merge --ff-only <remote>/<branch>` → `push <remote> <branch>`; on a non-ff push rejection it re-fetches+re-ffs and retries, bounded at `max_retries` push attempts total; returns `{"status": "pushed"|"conflict", "retried": n}`. NEVER composes `--force`/`-f`/`--force-with-lease` (asserted across every composed argv by `test_git_safe_push_never_composes_force`). Best-effort/never-raises, injected `run`/`sleep` seam for hermetic tests (`_FakeGitCompleted` fakes, no real git/network).
- **WU-2** `detect_cycle_bracket_friction` (`lazy_core/markers.py`) gained optional kwarg `concurrent_writer_commits: int | None = None` (@1707). In the signal-(b) `unexpected-commits` branch ONLY, a non-negative int makes the budget comparison use `max(0, commits_since - concurrent_writer_commits)`; `None`/absent ⇒ byte-identical to pre-WU-2 (the fail-safe/ambiguous path — an unknown concurrent count never suppresses a runaway). Guarded with `isinstance(...,int) and not isinstance(...,bool) and >=0`.
**SECONDARY (caller wiring — LANDED, wired + gated):** `_count_concurrent_writer_commits(repo_root, begin_head_sha)` added to `markers.py` (@2077) and wired into `cycle_end_friction_check` (@2226/2238) — it computes `concurrent_writer_commits` and passes it into `detect_cycle_bracket_friction`. Signal: a merge-excluded commit in `begin..HEAD` whose COMMITTER EMAIL differs from this repo's configured `git config user.email` is affirmative evidence it did not come from this cycle. Best-effort/never-raises; degrades to `None` (⇒ no suppression, the fail-safe default) on any git-read failure.
- **⚖ Documented signal limitation (honest, not a defect — READ BEFORE Phase 4):** this committer-email signal is CONSERVATIVE. A concurrent writer sharing THIS repo's SAME configured git identity — e.g. the operator's own second interactive/scheduled session, which is exactly the 2026-07-18 motivating incident's shape — commits under the identical email and is INVISIBLE to the signal: those commits correctly contribute 0 (never misattributed = the required fail-safe direction), so that specific same-identity incident is NOT suppressed by this signal alone. A genuinely distinct committer (a teammate, a bot, a differently-configured clone) IS caught today. Full attribution of same-identity concurrent SESSIONS wants the lane/lease commit-provenance the conflict-routing (Phase 4) + merge-back (Phase 5) phases establish — extend the caller there off lane provenance (the plan's "reuse WU-1's fetch context" pointer), not a new committer heuristic. Validation row 1 is satisfied at the `lazy_core` friction-detector-test level (detector suppresses when told) AND the caller is live for the distinct-committer case.
- **TERTIARY (LANDED):** added the `lazy-state.py` in-file `--test` fixture `concurrent-writer-commits-carve-out` (drives real `--cycle-begin`/`--cycle-end` subprocesses against a hermetic git repo with mixed committer identities) and regenerated the byte-pinned `tests/baselines/lazy-state-test-baseline.txt` via the sanctioned `_normalize_smoke_output` pipe (clean single-line diff).

### Phase: concurrent-worktree-agent-coordination P5 — Temp-worktree merge-back + cross-agent trailer

**Goal:** For a large/complex non-semantic conflict, complete the work in a temp worktree spun as a `lazy_coord` lane, merge back in queue order (abort-and-demote on conflict), and communicate to the conflicting agent via a `Concurrent-Merge-Back:` commit trailer — without halting the run.
**Entry criteria:** Phase 3 complete (lane/lease machinery), Phase 4 complete (the `classify_conflict` verdict gates this path — a `semantic` verdict halts and never reaches here).
**SPEC.md references:** Requirement 6; Locked Decisions 3–4; Validation row 5.

#### WU-1 — Merge-back lane wiring
- **Scope:** PHASES P5 deliverable "Merge-back orchestration wiring: on a `classify_conflict == write` large/complex verdict, spin a lane, complete the work, and merge_lane_branch in merge_order".
- **TDD:** yes.
- **Files to create/modify:** `user/scripts/lazy_coord.py` [VERIFY: `grep -n "^def merge_lane_branch\|^def merge_order\|^def lane_branch\|^def ledger_record_demotion" user/scripts/lazy_coord.py`] (+ any shared helper; if a helper must live in `lazy_core`, place it there — but the lane machinery stays in `lazy_coord.py`, which must not import `lazy_core`).
- **Test files:** the in-file `--test` harness in `user/scripts/lazy_coord.py`.
- **Test expectations:** (a) a lane spun for merge-back merges back in QUEUE order (`merge_order`), not completion order; (b) a merge conflict aborts-and-demotes with the lane branch PRESERVED and a `lanes.json` demotion record; (c) a clean merge records the merge in the ledger.
- **Implementation goal:** Wire the merge-back path onto the EXISTING lane primitives — reuse `lane_branch`/`lane_pool_dir` + `acquire_lease` to spin, `merge_order` for ordering, `merge_lane_branch` for the abort-and-demote merge. No new merge engine.
- **Spec requirements:** SPEC Requirement 6; Locked Decision 3.
- **Batch:** 1.

#### WU-2 — Concurrent-Merge-Back trailer compose/parse helper
- **Scope:** PHASES P5 deliverable "`Concurrent-Merge-Back:` trailer grammar + a helper that composes it into the merging commit and a reader that parses it from incoming history".
- **TDD:** yes.
- **Files to create/modify:** `user/scripts/lazy_core/runtimeplane.py` [VERIFY: `grep -n "^def _git\|^def git_head_short_sha" user/scripts/lazy_core/runtimeplane.py`] (compose/parse + the git read of incoming history), OR a small sibling in `lazy_core` if cleaner — cite the chosen file in the WU when landing.
- **Test files:** `tests/test_lazy_core/test_runtimeplane.py` [VERIFY: `test -f tests/test_lazy_core/test_runtimeplane.py`].
- **Test expectations:** a compose→parse round-trip recovers the affected paths + resolution guidance exactly; a commit message WITHOUT the trailer parses to empty (no false positive); the trailer format is a well-formed git trailer (`Concurrent-Merge-Back:` key).
- **Implementation goal:** Author the documented `Concurrent-Merge-Back:` trailer grammar (affected paths + one-line resolution guidance), a composer that appends it to the merging commit's message, and a parser that reads it from incoming history. Zero new contended state.
- **Spec requirements:** SPEC Locked Decision 4.
- **Batch:** 1 (disjoint from `lazy_coord.py`).

### Phase: concurrent-worktree-agent-coordination P2 — Git safety

**Goal:** Make every cycle git op safe under an open shared worktree (fetch+ff-before-push, bounded non-ff retry never force, pathspec-scoped commits) and stop the false process-friction signal on a sanctioned concurrent commit.
**Entry criteria:** Phase 1 complete (awareness prose establishes concurrent commits are expected).
**SPEC.md references:** Requirement 2 (Git safety); Validation Criteria rows 1–2.

#### WU-1 — Git-safety push helper
- **Scope:** PHASES P2 deliverable "New git-safety helper in runtimeplane.py: fetch + fast-forward + bounded non-ff push retry (never --force)".
- **TDD:** yes.
- **Files to create/modify:** `user/scripts/lazy_core/runtimeplane.py` [VERIFY: `grep -n "^def _git\|^def _current_head\|^def git_guard_status" user/scripts/lazy_core/runtimeplane.py`].
- **Test files:** `tests/test_lazy_core/test_runtimeplane.py` [VERIFY: `test -f tests/test_lazy_core/test_runtimeplane.py`].
- **Test expectations:** with an injected git `run`/`sleep` seam — (a) a push that races one non-ff rejection succeeds after fetch+ff+one retry; (b) `--force`/`-f` NEVER appears in any composed argv across all attempts; (c) the retry is bounded (an attempt cap; exhaustion returns a structured `conflict` result, not an infinite loop).
- **Implementation goal:** Add a helper alongside `_git`/`_current_head`/`git_guard_status` that fetches, fast-forwards, then pushes; on a non-ff rejection it re-fetches+ff and retries up to a bounded cap, never `--force`. Return a structured result (`pushed` / `retried: n` / `conflict`). Injected `run`/`sleep` seam (the `ensure_runtime` convention) for hermetic `--test`.
- **Spec requirements:** SPEC Requirement 2; Validation row 2.
- **Batch:** 1.

#### WU-2 — Process-friction carve-out for concurrent-writer commits
- **Scope:** PHASES P2 deliverable "`detect_cycle_bracket_friction` carve-out: a HEAD advance attributable to a concurrent writer does NOT append a process-friction unexpected-commits deny-ledger entry".
- **TDD:** yes.
- **Files to create/modify:** `user/scripts/lazy_core/markers.py` [VERIFY: `grep -n "detect_cycle_bracket_friction" user/scripts/lazy_core/markers.py`].
- **Test files:** `tests/test_lazy_core/test_markers.py` [VERIFY: `test -f tests/test_lazy_core/test_markers.py`], plus a `[process-friction]`-style fixture in `user/scripts/lazy-state.py`'s in-file `--test` harness [VERIFY: `grep -n "def test_" user/scripts/lazy-state.py | head`].
- **Test expectations:** (a) a commit whose author/session is a CONCURRENT writer (not this cycle's) does NOT append a `kind: process-friction` `unexpected-commits` entry to `lazy-deny-ledger.jsonl`; (b) a genuine runaway's own extra commit STILL appends the entry (existing behavior preserved); (c) an ambiguous case falls to the EXISTING friction behavior (fail-safe toward detecting runaways).
- **Implementation goal:** In `detect_cycle_bracket_friction`'s `unexpected-commits` branch, distinguish a sanctioned concurrent-writer HEAD advance (a commit not authored by this cycle's own dispatch — reuse the authorship/session signal established with WU-1's fetch context) from a runaway's own extra commits, and suppress the deny-ledger append only for the former. Conservative + fail-closed toward the current runaway-detection behavior. Shared `lazy_core` (both pipelines inherit; parity-audited).
- **Spec requirements:** SPEC Requirement 2; Validation row 1 (the motivating 2026-07-18 false-friction incident).
- **Batch:** 1.

### Phase: lazy-cycle-containment P2 — Cycle-subagent marker (C1)

**Goal:** Add the script-owned cycle-subagent context marker (`~/.claude/state/lazy-cycle-active.json`) read/write to `lazy_core.py` plus `--cycle-begin`/`--cycle-end` CLI on `lazy-state.py` and `bug-state.py` — the on/off switch every later layer keys on.

**Entry criteria:** part-1 Phase 1 complete (the `lazy_core.py` ↔ `lazy-state.py` CLI-surface pattern + `~/.claude/state/` marker conventions).

**SPEC.md references:** Technical Design §C1 (Cycle-subagent context marker); Validation Criteria row "Marker set/clear brackets a dispatch".

#### WU-1 — cycle marker helpers + `--cycle-begin`/`--cycle-end` CLI

**Scope:** PHASES.md P2 deliverables: `lazy_core.py` marker read/write/clear helpers; `--cycle-begin`/`--cycle-end` CLI with self-healing staleness + idempotent clear; mirror onto `bug-state.py`.

**TDD:** yes (WU-2 holds tests; implement to green).

**Files to create/modify:**
- `user/scripts/lazy_core.py` — `read_cycle_marker()`, `write_cycle_marker(...)`, `clear_cycle_marker()`.
- `user/scripts/lazy-state.py` — `--cycle-begin --feature-id <id> --nonce <hex> [--kind real|meta]` and `--cycle-end` flags.
- `user/scripts/bug-state.py` — mirror the two flags.

**Implementation goal:**
- The marker lives at `<state-dir>/lazy-cycle-active.json` where the state dir respects the existing `LAZY_STATE_DIR` override used by the run marker [VERIFY: grep -n "LAZY_STATE_DIR" user/scripts/lazy_core.py] (reuse the same state-dir resolver helper the run marker uses [VERIFY: grep -n "def.*state_dir\|state_dir(" user/scripts/lazy_core.py]). It is the sibling of the run marker `lazy-run-*.json` [VERIFY: grep -n "lazy-run" user/scripts/lazy_core.py].
- `write_cycle_marker(...)` writes the marker carrying: `feature_id`, `nonce`, `kind` (`real`|`meta`, default `real`), `started_at` (ISO 8601), parent `session_id` (best-effort from env, else null), and `commit_tally: 0`. Self-healing staleness: if a marker already exists, OVERWRITE it and log the event (the orchestrator is single-threaded — only one dispatch is ever in flight).
- `clear_cycle_marker()` deletes the marker; idempotent — a missing marker is a no-op exiting 0 with no error.
- `read_cycle_marker()` returns the parsed dict or `None` when absent. This is the single predicate Phase 3's refusals and the Phase-4 hook fast-path both consult.
- `--cycle-begin` calls `write_cycle_marker`; `--cycle-end` calls `clear_cycle_marker`. Mirror the two CLI flags onto `bug-state.py` (shared `lazy_core` backing — the flag handlers delegate to the same helpers) [VERIFY: test -f user/scripts/bug-state.py].

**Spec requirements:** SPEC §C1 — "A new `lazy-state.py --cycle-begin --feature-id <id> --nonce <hex> [--kind real|meta]` call ... Cleared by `lazy-state.py --cycle-end` ... Idempotent ... Carries: `feature_id`, dispatch `nonce`, `kind`, `started_at`, parent `session_id`, and a `commit_tally` ... Self-healing staleness: if a `--cycle-begin` finds an existing cycle marker ... it is overwritten and the event logged".

**Batch:** 1

**Goal:** `lazy_core.py` refuses the orchestrator-only state-script operations when the cycle marker is present — the belt-and-suspenders backstop if the C2 hook is disabled/bypassed. Lands BEFORE the hook (part-3 Phase 4).

**Entry criteria:** Phase 2 complete (`read_cycle_marker()` exists).

**SPEC.md references:** Technical Design §C3 (Refuse-by-construction); Validation Criteria rows "Orchestrator-only op refuses under marker", "Same ops allowed without marker".

#### WU-3 — refuse-by-construction guards on orchestrator-only ops

**Scope:** PHASES.md P3 deliverables: refusals on `--run-end`, `--run-start`, `--apply-pseudo`, `--enqueue-adhoc`, `--emit-dispatch` when the marker is present; allow-listed ops + reads continue to work; invariant comments.

**TDD:** yes (WU-4 holds tests).

**Files to create/modify:**
- `user/scripts/lazy_core.py` — refusal guard at the entry of each orchestrator-only op + invariant comment block.
- `user/scripts/lazy-state.py` / `user/scripts/bug-state.py` — ensure the refusal fires on the corresponding CLI flags (the guard lives in `lazy_core.py`, surfaced through both).

**Implementation goal:**
- Add a guard helper (e.g. `refuse_if_cycle_active(op_name)`) that, when `read_cycle_marker()` is not `None`, prints a corrective message to stderr and exits non-zero with ZERO side effects. Invoke it at the entry of the handlers for `--run-end` [VERIFY: grep -n "run-end\|def.*run_end\|delete_run_marker" user/scripts/lazy_core.py], `--run-start` [VERIFY: grep -n "run-start\|def write_run_marker" user/scripts/lazy_core.py], `--apply-pseudo` [VERIFY: grep -n "apply-pseudo\|apply_pseudo" user/scripts/lazy_core.py], `--enqueue-adhoc` [VERIFY: grep -n "enqueue-adhoc\|enqueue_adhoc" user/scripts/lazy_core.py], `--emit-dispatch` [VERIFY: grep -n "emit-dispatch\|emit_dispatch" user/scripts/lazy_core.py].
- The guard must fire BEFORE any side effect (marker write/delete, queue mutation, prompt emission) so a refused op leaves state untouched.
- Allow-listed ops (`--neutralize-sentinel`, `--verify-ledger` [VERIFY: grep -n "neutralize-sentinel\|verify-ledger" user/scripts/lazy_core.py]) and all read/probe ops are NOT guarded — they remain callable with the marker present (a legitimately-dispatched subagent needs them).
- Add an invariant comment: the orchestrator sets the marker → dispatches → clears on return → only THEN runs these ops, so the refusal bites only a subagent calling them mid-dispatch.
- The refused-op set here MUST match the C2 hook's loop-formation/lifecycle deny-set (part-3 Phase 4) — they are intentionally redundant defense-in-depth.

**Spec requirements:** SPEC §C3 — "`--run-end`, `--run-start`, `--apply-pseudo`, `--enqueue-adhoc`, `--emit-dispatch` REFUSE (exit non-zero, zero side effects, corrective message) when `lazy-cycle-active.json` is present ... Safe for the orchestrator by construction".

**Batch:** 2

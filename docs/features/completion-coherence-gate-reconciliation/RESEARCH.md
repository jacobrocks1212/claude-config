Reconciling Redundant Completion Gates in Autonomous AI Pipelines: A Research Report on Evidence-Gated Coherence and State-File Mutation
Executive Summary
The transition of an autonomous AI software-development pipeline from heuristic-based completion checks to a strictly evidence-gated state machine requires resolving epistemological conflicts between redundant "definition of done" gates. In the evaluated architecture, the downstream checker (check-docs-consistency.ts) relies on a rigid physical evaluation of a Markdown-based plan file (PHASES.md), while the Python-driven execution harness utilizes runtime artifacts (VALIDATED.md, MCP_TEST_RESULTS.md) as the ultimate cryptographic source of truth. Implementing the locked "Direction A"—extending the verification-only carve-out to the completion gate, gating it on on-disk passing evidence, and auto-ticking certified verification rows—reconciles these disparate systems. It achieves this by normalizing the human-readable plan file to match the verifiable on-disk evidence, thereby eliminating the redundant coherence-recovery cycle that wastes language model compute.
The exhaustive analysis of industry prior art in CI/CD pipeline automation, Supply-chain Levels for Software Artifacts (SLSA), and in-toto attestation frameworks yields the following high-leverage recommendations for the safe implementation of the auto-tick rewrite mechanism:
1. Adopt a Verification Summary Attestation (VSA) Model for Evidence Evaluation: The pipeline must treat MCP_TEST_RESULTS.md as the raw test execution payload and VALIDATED.md as the authoritative attestation envelope. The completion gate must evaluate the union of these documents, demanding strict topological coherence—specifically ensuring that the ratio of passed tests equals the total, and that the total is greater than zero—before authorizing any state mutation.
2. Enforce Strict Time-of-Check-to-Time-of-Use (TOCTOU) Protections for HEAD Drift: To prevent the certification of stale or altered code, the gate must refuse completion if the validated_commit pinned in the evidence does not match the current repository HEAD. The singular exception to this rule should be a limited "auto-revalidate" or warning mechanism applied strictly to non-executable documentation diffs, thereby preventing malicious or accidental state mutations from bypassing the runtime validation gate.
3. Implement Fully Atomic, POSIX-Compliant File Operations for State Mutation: The automated mutation of PHASES.md must avoid partial writes or race conditions by utilizing a strict write-to-temp-file methodology. This requires leveraging Python's os.replace combined with explicit buffer flushing and os.fsync (or fcntl.F_FULLFSYNC on macOS) to ensure absolute data durability prior to the atomic rename of the state file.
4. Inject Auditable Traceability into the Mutated State File: The auto-tick mechanism must leave a deterministic, byte-stable audit trail directly within the Markdown file. Modified checkboxes should be appended with an invisible HTML comment (e.g., <!-- auto-ticked: validated_commit=HEAD -->) to prevent the loss of data lineage and to clearly distinguish automated gate mutations from manual human edits or upstream LLM agent generations.
5. Deploy Dual-Layer Validation Integrating Linting and Evidence Gating: Checklist-drift friction cannot be durably solved by evidence-gating alone. A robust pipeline requires a "Swiss Cheese" model of defense that utilizes authoring-time linting to enforce structural marker correctness, paired with an execution-time gate that honors the on-disk evidence. This prevents the silent authorization of implementation rows that a hallucinating LLM agent may have incorrectly marked as verification-only.
Contextual Analysis: Redundant Gates and State Consistency
In autonomous agentic software development, the coordination between non-deterministic language models and deterministic execution harnesses frequently suffers from state-synchronization friction. The system in question utilizes a script-owned state machine (lazy_core.py / lazy-state.py) that drives progress through rigid, sequential phases (/spec through /mcp-test). The plan file (PHASES.md) serves as the central ledger, tracking both implementation deliverables and verification checks.
The core defect arises from divergent completion criteria embedded within the system's verification gates. Upstream gates, such as remaining_unchecked_are_verification_only, successfully recognize the <!-- verification-only --> structural marker and exempt these rows from manual ticking requirements. However, the terminal completion gate (_phase_completion_plan) and an uneditable downstream TypeScript checker (check-docs-consistency.ts) demand absolute physical coherence, requiring every - [ ] to be explicitly marked as - [x]. This disagreement forces the LLM agent into a redundant "coherence-recovery" cycle. The agent is forced to manually edit the Markdown file to tick boxes that the on-disk evidence (VALIDATED.md) already proves are complete.
The locked "Direction A" resolves this architectural tension by dynamically applying the exemption at completion time, contingent on the presence of passing on-disk evidence, and subsequently auto-mutating the PHASES.md file to satisfy the downstream checker. This approach mirrors artifact normalization patterns found in mature CI/CD ecosystems, where a trusted intermediary processes raw execution data and normalizes it into a universally consumable format for downstream systems that lack nuanced policy evaluation logic.
Per-Question Findings
1. The Authoritative Evidence Set
The treatment of multiple evidence files (VALIDATED.md and MCP_TEST_RESULTS.md) must be rigorously defined to prevent authorization bypasses and ensure pipeline integrity. In the context of supply chain security and the SLSA framework, this relationship is highly analogous to the interaction between raw provenance data and a Verification Summary Attestation (VSA).
The raw test output (MCP_TEST_RESULTS.md) acts as the execution provenance, containing the quantitative metrics (pass == total, result: all-passing). The VALIDATED.md receipt acts as the VSA—a higher-level assertion that the underlying execution metrics meet the pipeline's policy requirements. In systems like in-toto, the attestation consists of an envelope (the authentication wrapper) and a predicate (the actual claim or test result).
The gate must require both files to exist and be internally coherent, establishing a strict precedence order where VALIDATED.md acts as the primary trigger, but its cryptographic and logical validity is entirely contingent on the data within MCP_TEST_RESULTS.md. Accepting either file in isolation degrades the trust model significantly. If VALIDATED.md is accepted without the underlying test results, the system becomes vulnerable to forged assertions where a receipt is minted without actual test execution. Conversely, if MCP_TEST_RESULTS.md is accepted without the VSA, the system bypasses the formal policy-evaluation phase.
Before treating the evidence as authoritative, the gate must execute a series of deterministic checks. It must parse MCP_TEST_RESULTS.md and verify that the result field strictly equals all-passing. Furthermore, it must mathematically verify that pass == total and, crucially, that total is greater than zero. A test suite that successfully passes zero tests is a widely recognized CI/CD anti-pattern that can lead to false-positive completions, silently approving unimplemented features. Finally, the frontmatter of VALIDATED.md must contain kind: validated and an explicit reference to the commit hash (validated_commit == HEAD), ensuring the attestation is bound to the current state of the repository.
2. Resolving Skip and Deferred Edge Cases
When the runtime test step yields a SKIP_MCP_TEST.md or a DEFERRED_* receipt instead of passing evidence, the completion gate faces a critical decision regarding policy relaxation. Continuous integration promotion gates, such as those implemented in Argo, Spinnaker, and Chainloop, typically operate on a strict "fail closed" principle. In these ecosystems, a skipped test is functionally and securely equivalent to an absent or failed test unless it is explicitly authorized by a recognized human or cryptographic authority.
The completion gate must refuse the auto-tick exemption when a skip or deferral receipt is encountered. Exempting the row based merely on the presence of a skip receipt would fundamentally undermine the definition of done, allowing features to be marked complete without runtime verification. The only acceptable condition for exemption in the presence of a skip receipt is if the receipt contains an explicit, cryptographically verifiable operator signature or a rigid policy-override marker generated by an authorized system administrator.
In advanced compliance frameworks like the Compliance Data Lakehouse or Palantir's in-toto implementations, attestations are still recorded for skipped phases to maintain a continuous audit trail, but the policy engine explicitly marks them as "not compliant" or "skipped," which actively blocks downstream deployment. If the autonomous pipeline requires the ability to defer tests, the gate must not auto-tick the verification rows. Instead, the optimal pattern is to inject a specific <!-- deferred --> HTML comment into the Markdown file, leaving the physical checkboxes blank. This preserves the visual indication that work remains unfinished while allowing the state machine to transition to a deferred state, provided the downstream checkers are updated to recognize the deferral. Because the downstream TypeScript checker in this specific architecture cannot be edited and mindlessly counts all checkboxes, the only mathematically sound approach is to refuse completion until the test is either successfully run or the deliverables are manually excised from the plan by the LLM agent.
3. Managing Freshness and HEAD Drift
The evidence receipt pins the validation to a specific state of the repository via the validated_commit == HEAD assertion. If a commit lands between the generation of the receipt in the /mcp-test phase and the execution of the completion gate, a Time-of-Check-to-Time-of-Use (TOCTOU) vulnerability manifests. In this scenario, the code that was validated is no longer the code being promoted, entirely invalidating the attestation.
In strict attestation-based promotion models, such as SLSA Build Level 3, attestations are immutably bound to the exact Git commit or artifact digest they certify. Therefore, if validated_commit no longer equals the current repository HEAD at completion time, the primary and safest action is for the gate to refuse and require revalidation.
However, autonomous agentic pipelines present a unique nuance: the PHASES.md plan file itself is frequently updated by the agent to track progress. Strict HEAD checking will consistently fail if the agent commits a documentation or status update after the test run but before the completion gate triggers. To mitigate this friction without compromising the security posture, the gate should inspect the Git diff between validated_commit and the current HEAD. If the diff reveals that only non-executable documentation files (e.g., PHASES.md, README.md) have been altered, the gate may safely warn and proceed, auto-ticking the file. Conversely, if the diff contains any source code, executable scripts, or configuration files that could alter runtime behavior, the gate must immediately refuse completion and force a re-run of /mcp-test.
4. Safety Mechanisms for Auto-Mutation of State Files
Automated gates that mutate the very source-of-truth files they are responsible for evaluating carry severe risks of data corruption, partial writes, and the loss of auditability. Modifying PHASES.md in place requires strict guarantees of both idempotency and atomic file system operations.
To guarantee crash-safety, the Python rewrite logic must completely avoid naive open('w') or open('r+') operations, which can truncate the file if the script crashes mid-execution. Instead, the implementation must strictly utilize a write-to-temporary-file pattern. The updated content is written to a temporary file in the exact same directory (e.g., PHASES.md.tmp), followed by an explicit fsync command to flush the operating system's buffers to physical disk. Finally, an atomic os.replace operation is executed to overwrite the original file. On POSIX-compliant systems, os.replace guarantees that the file is updated atomically; external readers will either see the entirely old file or the entirely new file, never a corrupted, partially written middle state. It is critical to note that on macOS, the standard fsync call may not flush internal hardware buffers; utilizing fcntl.F_FULLFSYNC is recommended to ensure absolute data durability.
Beyond crash-safety, the mutation must be idempotent and highly auditable. The - [ ] to - [x] translation must be anchored specifically to the <!-- verification-only --> marker using tightly constrained regular expressions. These expressions must enforce line-level boundaries to prevent the accidental mutation of similarly named strings embedded within documentation or code blocks. Furthermore, to establish a clear audit trail that distinguishes the automated gate's action from an LLM agent's manual action, the rewrite logic should append a byte-stable inline HTML comment to the modified row. By modifying the line to read - [x] Verify API endpoint response <!-- verification-only --> <!-- auto-ticked: validated_commit=abc1234 -->, subsequent audits can immediately identify the exact commit and programmatic actor responsible for ticking the box, effectively mitigating the "who ticked this and why" ambiguity that plagues automated checklists.
5. Downstream-Checker Confirmation and Artifact Normalization
The architecture dictates that a downstream, non-editable TypeScript script (check-docs-consistency.ts) executes post-flip. This script evaluates the absolute coherence of the file, counting every checkbox without honoring any upstream carve-outs or semantic markers.
The auto-ticking mechanism proposed in Direction A is fundamentally sufficient to satisfy a "count everything" checker. Because the downstream checker operates on a naive text-parsing logic—evaluating the physical - [x] state rather than semantic intent—modifying the upstream file to physically reflect completion acts as a standard artifact-normalization pipeline. By rewriting the - [ ] into - [x] prior to the status flip to Complete, the Python execution harness acts as an evidence-backed gatekeeper, normalizing the artifact and shielding the downstream TypeScript script from the complexity of the carve-out logic.
However, realistic residual edge cases exist where coherence will still break if the normalization pass is not robust:
1. Superseded Phases: If a phase was marked as superseded or abandoned but still contains unchecked deliverables, the downstream checker will fail. The Python auto-tick pass must either explicitly prune unchecked boxes from phases marked Superseded, or the authoring-time linting tool must force the LLM to remove them.
2. Zero-Deliverable Phases: If a phase contains no checkboxes at all, standard dumb checkers may fail if they enforce a count > 0 validation to ensure that a phase actually accomplished work. The Python gate should ensure that every phase has at least one valid deliverable or a structural placeholder before declaring the plan complete.
3. Non-Checkbox Status Lines and Stray Brackets: If the downstream checker parses the entire document rather than explicitly targeting markdown lists, any stray [ ] brackets in paragraph text might trigger a failure. The Python normalization step should strictly target lines matching the standard Markdown list-item regex (^\s*-\s+\[\s\]) to avoid accidentally modifying documentation content.
6. Balancing Lint-up-Front versus Evidence-Gated Promotion
The tension between enforcing correct Markdown authoring via linting upfront and honoring runtime evidence at the gate represents a classic architectural tradeoff. The recurring "tick the boxes" friction prompts the question of whether authoring-time enforcement or execution-time evaluation is superior. From the perspective of mature CI/CD ecosystems and compliance frameworks, neither approach is sufficient in isolation; they must be deployed in tandem.
The durable fix requires both authoring-time linting (to force correct row marking) and the evidence-based gate (to auto-tick the rows). Relying solely on linting fails because the LLM agent is still forced into the redundant coherence-recovery cycle; the agent will successfully author the markers, but will still be blocked at completion until it manually ticks them, perpetuating the compute waste. Conversely, relying solely on the gate honoring the marker introduces severe security risks. A hallucinating LLM might attach the <!-- verification-only --> marker to a critical implementation deliverable (e.g., - [ ] Write database migration <!-- verification-only -->). The gate, seeing passing test evidence for the unrelated tests, would blindly auto-tick the unwritten migration, silently passing incomplete code into production.
Therefore, authoring-time linting must structurally enforce that verification markers are only applied to rows that match the semantic profile of a test (e.g., verifying the deliverable string against a known list of test commands or utilizing PyMarkdown AST analysis). The completion gate then handles the physical state mutation based on the cryptographic evidence, creating a highly resilient, multi-layered defense.
7. Guardrails Against Over-Relaxation
To prevent an evidence-gated exemption from silently passing features with genuine unfinished work, the pipeline must employ defense-in-depth data cross-validation, mirroring the attestation verification models dictated by SLSA and in-toto.
The auto-tick logic must utilize a strict cardinality check to prevent over-relaxation. The gate must count the absolute number of - [ ] verification-marked rows slated for auto-ticking and compare this integer against the pass metric extracted from MCP_TEST_RESULTS.md. If the auto_tick_count is strictly greater than the pass_count, the gate must refuse the operation. This mathematical imbalance indicates that there are more verification deliverables claimed in the plan than actual tests executed by the runtime harness, signaling either a marker-drift hallucination by the LLM or stale, forged test evidence.
Additionally, the Python gate must verify the structural integrity of the PHASES.md file. Instead of relying purely on naive substring replacement, the script should ideally utilize an AST-aware parser (such as mistletoe or marko) or a highly constrained regex to ensure the <!-- verification-only --> marker is legitimately part of a list item and not embedded in a code block, comment, or blockquote.
8. Reversibility and Kill-Switch Mechanisms
Shipping a gate-relaxation logic into a deterministic state machine requires a frictionless rollback mechanism. If the auto-tick logic proves too permissive, introduces race conditions, or interacts poorly with unforeseen LLM agent drift, operators must be able to disable the relaxation without reverting the core Python codebase or waiting for a CI/CD build cycle.
The standard industry convention for shipping critical pipeline modifications is to gate the relaxation behind a documented, explicit environment variable (e.g., LAZY_STRICT_EVIDENCE_GATE=1 or LAZY_DISABLE_AUTOTICK=1). The granularity of this kill-switch should be implemented directly at the feature-flag level within lazy_core.py. If the environment variable is toggled to true, the script immediately bypasses the VALIDATED.md evaluation, falls back to the legacy _phase_completion_plan logic (which rigidly demands all boxes be manually ticked by the agent), and entirely skips the file mutation phase. This ensures graceful degradation of the pipeline back to its proven, albeit less efficient, baseline state.

	Auto-Tick Implementation Checklist
To guarantee safe, idempotent, and auditable mutation of the human-readable PHASES.md file, the implementer must ensure the Python logic strictly adheres to the following sequence. This checklist addresses POSIX filesystem quirks, regex safety, and semantic preservation.
1. Crash-Safety & Atomic I/O
* [ ] Temp File Initialization: Open a temporary file in the exact same directory as the target file (e.g., PHASES.md.tmp) to guarantee it resides on the same filesystem, preventing cross-device link errors during rename.
* [ ] Stream Writing: Stream the modified lines to the temporary file sequentially rather than loading and holding the entire string in memory, mitigating large-file truncation risks during resource constraints.
* [ ] Buffer Synchronization: Invoke f.flush() and os.fsync(f.fileno()) before closing the temporary file to ensure operating system write buffers are fully committed to physical persistent storage. (Note: Utilize fcntl.F_FULLFSYNC if executing on macOS hardware).
* [ ] Atomic Swap: Utilize os.replace(tmp_path, final_path) to atomically overwrite the original file, guaranteeing that a sudden SIGKILL does not leave an empty or corrupted PHASES.md.
2. Idempotency & Regex Constraints
* [ ] Anchored Matching: The regular expression utilized for substitution must anchor to the start of the line, specifically matching ^\s*-\s+\[\s\] to avoid altering inline text or Markdown code blocks.
* [ ] Marker Verification: The regular expression must require the explicit presence of the <!-- verification-only --> string on the exact same line as the target checkbox.
* [ ] Idempotent Execution: The rewrite logic must be strictly idempotent. If the script is run twice consecutively against the same file, the file hash must not change on the second execution.
3. Auditability & Traceability
* [ ] Audit Trail Injection: When transforming - [ ] to - [x], append an inline, byte-stable HTML comment logging the mutation. Example: <!-- auto-ticked: commit=abc1234 -->.
* [ ] Receipt Logging: Record the absolute number of ticked rows directly into the final COMPLETED.md receipt, explicitly stating that the completion gate invoked the auto-tick mechanism rather than the LLM agent.
4. Semantic Drift Protection
* [ ] Exclusion of Code Blocks: Ensure the parser actively ignores lines wrapped inside triple backticks (```) to prevent mutating documentation that happens to describe or provide examples of checklists.
* [ ] Cardinality Lock: Assert that the metric Total Ticked Rows is less than or equal to Total Tests Passed. Raise a terminal runtime exception if this invariant is violated.

Final Verdict: The auto-ticking mechanism of Direction A is fully sufficient to satisfy the downstream checker, provided the normalization pass is absolute and exhaustive.
Because the downstream checker operates on a naive text-parsing logic—evaluating physical - [x] states rather than semantic intent—modifying the upstream file to physically reflect completion acts as a standard, highly effective artifact-normalization pipeline. By rewriting the - [ ] into - [x] prior to the status flip to Complete, the Python execution harness acts as an evidence-backed gatekeeper. It successfully shields the downstream TypeScript script from the complexity of the carve-out logic, ensuring that the downstream system receives the pristine, fully checked artifact it expects without requiring modifications to the TypeScript codebase.
Residual Cases Requiring Explicit Handling: While the auto-tick satisfies the active verification rows, coherence breaks will still occur in the following edge cases if not addressed by linting or the normalization script:
1. Superseded Phases: If an earlier phase in PHASES.md was abandoned but retains unchecked boxes, the TypeScript script will fail. The Python auto-tick pass must either explicitly prune unchecked boxes from phases structurally marked as Superseded, or the upstream linting tool must force the LLM to remove them entirely during the phase transition.
2. Malformed Checkboxes: If the LLM generates a checkbox with invalid or irregular space characters (e.g., - [ ]), the Python regex might miss it, leaving it untouched. The downstream TypeScript script will subsequently flag this as incomplete. The Python regex must be designed to account for variable internal whitespace (^\s*-\s+\[\s+\]).
3. Marker Drift: If the <!-- verification-only --> string contains a typo or capitalization error (e.g., <!-- verification-onlly -->), the Python auto-tick will skip the row, resulting in a downstream failure.

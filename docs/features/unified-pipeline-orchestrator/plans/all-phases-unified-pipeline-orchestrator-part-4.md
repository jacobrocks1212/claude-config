## Phase: unified-pipeline-orchestrator P4 — Toolify miner + deterministic-only bar

**Goal:** Ship `toolify-miner.py` (stdlib-only, read-only over logs) ranking recurring deterministic tool-call sequences as toolify candidates, plus the deterministic-only-bar doc (candidate schema + promotion checklist). Miner proposes; promotion stays deliberate.

**Entry criteria:** None — independent of Phases 1–3 (sequenced after them per SPEC numbering, but no code dependency).

**SPEC.md references:** "Toolification framework" UX + Technical Design ("Miner", "The deterministic-only bar", "Promotion is deliberate"); Implementation Phase 4; Validation Criteria "Miner ranks real candidates"; Open Question "Miner signature granularity"; RESEARCH_SUMMARY Locked Decisions 3 & 5.

### WU-1 — `toolify-miner.py` parser + signature normalization + ranking

- **Scope:** Deliverable "`toolify-miner.py` (stdlib-only): parses `~/.claude/projects/**/*.jsonl` (+ `subagents/agent-*.jsonl`), extracts orchestrator-turn tool-call sequences, normalizes into signatures (tool + argument-shape, values elided), ranks by `occurrences × est_tokens_per_occurrence`; emits markdown table + JSON; never mutates logs".
- **TDD:** yes.
- **Files to create/modify:**
  - `user/scripts/toolify-miner.py` — net-new, stdlib-only (no third-party imports). Argparse `--logs <dir>` (default `~/.claude/projects`), `--json`/`--markdown` output selection.
- **Test files:** `user/scripts/test_toolify_miner.py` — net-new.
- **Test expectations:** parses fixture `.jsonl` transcripts; extracts tool-call sequences from orchestrator turns; normalizes two occurrences of the same dance (different argument VALUES, same argument SHAPE) to ONE signature; ranks by the documented score; emits a markdown table + JSON; the fixture log dir hash is unchanged after the run (read-only proof).
- **Implementation goal:** A signature is `(tool_name, argument-shape)` with values elided; ranking = `occurrences × est_tokens_per_occurrence`. Output is both a markdown table and JSON. Read-only: open log files in read mode only; never write/rename/delete under the logs dir.
- **Spec requirements:** the "Miner" Technical Design bullet verbatim.
- **Batch:** 1.

### WU-2 — Deterministic-only bar (classifier) + signature-granularity tuning

- **Scope:** Deliverable "Deterministic-only bar applied: candidate surfaces above the bar iff deterministic AND repeated AND token-heavy; judgment steps (verdicts, recovery dispatch, `--verify-ledger`) explicitly out of scope and rank below" + "Open Question resolved: signature granularity tuned against real logs".
- **TDD:** yes.
- **Files to create/modify:**
  - `user/scripts/toolify-miner.py` — the bar classifier: a candidate is above-bar iff (a) deterministic (branches computable from observable state, not agent reasoning — heuristic: the sequence contains no judgment-marker tool calls such as `AskUserQuestion`, no verdict/recovery-dispatch step), (b) repeated (occurs across ≥2 runs), (c) token-heavy (score exceeds a documented threshold). Judgment sequences rank below.
- **Test files:** `user/scripts/test_toolify_miner.py`.
- **Test expectations:** a fixture deterministic dance (×N across ≥2 runs) classifies above-bar; a fixture judgment sequence (contains a verdict/recovery step) classifies below-bar even if frequent; the `--verify-ledger`-shaped sequence is below-bar. **Granularity tuning (closes the Open Question):** the chosen signature coarseness clusters "the same dance" across runs without over-merging two genuinely distinct sequences — assert both a positive (two value-variant occurrences merge) and a negative (two shape-distinct sequences do NOT merge) case. Record the chosen coarseness + rationale in the WU's Implementation Notes and in the bar doc (WU-3).
- **Implementation goal:** The bar is a deterministic + repeated + token-heavy AND-gate; judgment steps are excluded by construction.
- **Spec requirements:** "The deterministic-only bar" Technical Design bullet; Open Question "Miner signature granularity".
- **Batch:** 2 (extends WU-1's `toolify-miner.py` — sequential).

### WU-3 — Bar/schema/promotion-checklist doc + miner tests finalization

- **Scope:** Deliverable "Bar/schema/promotion-checklist doc: candidate schema fields + the deliberate-promotion checklist (miner proposes → reviewed change → optional future harden-harness `/spec-bug` auto-initiation)" + the "Miner ranks real candidates" Runtime Verification row.

- **Implementation goal:** The doc defines the schema + bar + promotion checklist; it documents that promotion is deliberate (the miner never auto-writes code). The Runtime Verification row (running the miner over REAL logs surfaces the three retro-named dances on top) is a manual/integration check, NOT closed by `/execute-plan`.
- **Spec requirements:** "Promotion is deliberate. The miner proposes; … The framework defines the candidate schema and the promotion checklist; it does not auto-write code."
- **Batch:** 2 (parallel with WU-2? — NO: WU-3 documents WU-2's bar + tuning, so it follows WU-2). Place WU-3 in **Batch 3**.

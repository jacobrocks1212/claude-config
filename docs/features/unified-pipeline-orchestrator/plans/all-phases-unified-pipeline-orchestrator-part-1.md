## Phase: unified-pipeline-orchestrator P1 — Merged work-list view + ordering

**Goal:** Add a thin, stdlib-only merged work-list view that orders both queues (priority desc; bug breaks ties) and returns the next actionable item as `{item_id, type, repo_root}`, without re-inferring per-item state.

**Entry criteria:** None (first phase).

**SPEC.md references:** "Unified orchestrator — one skill, two state scripts, merged work-list (chosen architecture)"; Validation Criteria rows "Merged run drains both types" and "Single-type unchanged"; Open Question "Ordering field source"; RESEARCH_SUMMARY Locked Decisions 1 & 2.

### WU-1 — Merged-view helper + ordering comparator (`--next-merged`)

- **Scope:** Deliverables "merged-view helper (`lazy-state.py --next-merged`)" + "ordering rule implemented (priority/tier desc; tie → bug before feature; stable)" + "reuses existing queue loaders + binds active repo".
- **TDD:** yes.
- **Files to create/modify:**
  - `user/scripts/lazy-state.py` — add `--next-merged` argparse arg + dispatch in `main()` [VERIFY: grep -n "def main" user/scripts/lazy-state.py]; bind active repo via `lazy_core.set_active_repo_root` [VERIFY: grep -n "def set_active_repo_root" user/scripts/lazy_core.py] before reading queues.
  - `user/scripts/lazy_core.py` — add the merged-view helper here (shared by both scripts), reusing the existing feature/bug queue loaders rather than re-parsing JSON.
- **Test files:** `user/scripts/test_lazy_core.py` (characterize the merged-view helper directly).
- **Test expectations:** both queues populated → ordered list head correct; equal priority → `type == "bug"` precedes `type == "feature"`; only-features → identical to `lazy-state.py` single-current head; only-bugs → identical to `bug-state.py` head; both empty → no item / None. Each test must FAIL with the helper absent (RED-for-the-right-reason: import/attribute miss is acceptable RED here since the symbol does not yet exist, but the assertion must encode the ordering contract, not the absence).
- **Implementation goal:** The helper reads `docs/features/queue.json` and `docs/bugs/queue.json` via the existing loaders [VERIFY: grep -n "queue.json" user/scripts/lazy-state.py], applies the comparator (priority/tier desc, bug-breaks-tie, stable for equal (priority, type)), and returns `{item_id, type, repo_root}` for the head. It MUST NOT call `compute_state` or otherwise re-infer per-item state. Output shape is JSON on stdout for `--next-merged`.
- **Spec requirements:** "the layer returns `{item_id, type: feature|bug, repo_root}` honoring priority (bugs break ties)"; "It does not re-infer per-item state — it asks each state script for that."
- **Batch:** 1.

### WU-2 — Resolve ordering-field source (priority/tier normalization)

- **Scope:** Deliverable "Resolves the Open-Question ordering field source — confirm whether feature/bug `queue.json` items carry a comparable `tier`/`priority` field or need a normalization map" + the Runtime Verification spike row "ordering-field ground truth".
- **TDD:** yes.
- **Files to create/modify:**
  - `user/scripts/lazy_core.py` — the comparator's priority-extraction logic (a small normalization map IFF the spike shows feature/bug items use different field names/scales).
- **Test files:** `user/scripts/test_lazy_core.py`.
- **Test expectations:** given a feature item with `{tier: 1}` and a bug item with the bug-queue's actual priority field at equal effective priority, the comparator orders the bug first; given differing priorities, higher wins regardless of type.
- **Implementation goal:** **Runtime spike requirement (closes the spike row):** before locking the comparator, READ a REAL `docs/features/queue.json` and `docs/bugs/queue.json` and record the OBSERVED field names + value ranges in the WU's Implementation Notes (the evidence that closes the spike). A static read of the enqueue code does NOT satisfy this — inspect actual on-disk queue files. Then implement the extraction: if both queues share a comparable field, use it directly; if not, add a documented normalization map. Cite the observed fields in `PHASES.md`'s `## Validated Assumptions` block.
- **Spec requirements:** Open Question "Ordering field source. Do feature/bug `queue.json` items already carry a comparable `tier`/`priority`, or does the merged view need a normalization map? (verify during Phase 1)".
- **Batch:** 1 (same file as WU-1 in `lazy_core.py` — see File-Overlap note below; runs as Batch 2 to avoid the overlap).

## Blocking Issue Protocol

If a blocking issue is encountered:
1. Stop all in-progress work; launch no new subagents.
2. Commit and push any completed work.
3. Print a blocking-issue report (completed work + commit, blocked WU, reason, recovery suggestion, remaining WUs).
4. Do not work around the blocker.

Blocking issues: a subagent failure unfixable after 2 retries; a quality-gate failure unfixable after 2 retries; a git push conflict unresolvable by rebase; any error requiring architectural decisions beyond the specs (e.g. the ordering-field spike reveals the two queues' priority semantics are genuinely incompatible and a normalization map is a product decision — write `NEEDS_INPUT.md` and halt).

## Key design contract (read before WU-1.1)

- **Capture is ONE call site** inside the shared `apply_pseudo` `__mark_complete__`/`__mark_fixed__`
  branch — both pipelines inherit it (parity by construction). It runs AFTER the receipt write,
  is eligibility-gated (`"interventions": true` top-level in `docs/features/queue.json` OR a
  present `## Intervention Hypothesis` block), attaches `intervention_recorded`/`intervention_record`
  keys ONLY when it fires (byte-identity otherwise), and is FAIL-OPEN to `warnings` (a capture
  error never fails a completion).
- **One serializer** (`lazy_core._render_intervention_record`) owns the record's frontmatter
  field order; capture writes it and the evaluator re-writes through it (diff-stable).
- **Run identity** = telemetry `run_id` (ISO `%Y-%m-%dT%H:%M:%SZ` — lexical == chronological).
  Baseline freezes `last_run_id`; the post window is distinct run_ids strictly greater.
  Review k (0-based) uses the non-overlapping post-run slice `[k·R, (k+1)·R)`; due when
  total post runs ≥ `(k+1)·R`.
- **Verdict bands (D5-A):** rates are events-per-run. `rel = (post−base)/base·100`;
  `base == 0 ∧ post > 0` ⇒ rel = +100. decrease: rel ≤ −band → CONFIRMED, rel ≥ +band →
  REFUTED, else INCONCLUSIVE; increase mirrored. min-sample: baseline+post events ≥ 5 or
  INCONCLUSIVE. All four numbers per-record overridable via the hypothesis block.
- **HARD:** all new `test_lazy_core.py` functions appended to `_TESTS`; all writes via
  `_atomic_write`; diagnostics via `_diag`; never print from lazy_core.

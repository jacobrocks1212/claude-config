### Phase: byref-dispatch-undercounts-forward-cycles P1 — Wire the monotonic state-change advance into the real-skill probe path

**Goal:** Make real-skill (by-reference) dispatch cycles advance `forward_cycles` via the consume-INDEPENDENT `advance_forward_cycle` (the `[feature_id, current_step, sub_skill]` state-change trigger), so the forward count no longer depends on the non-monotonic `consumed_emission_count()` census. This defeats BOTH the `advance_meta_cycle` `+1` over-absorb (Contributor A) and ring-cap census regression (Contributor B) at once.

**Entry criteria:** None (first phase).

**SPEC.md references:** `## Fix Scope` item 1 (Primary); `## Proven Findings` (root cause + "Why the robust trigger doesn't save it"); Symptom 3 (the `--repeat-count` path advances only via `advance_run_counters`).

  - Add `lazy_core.advance_forward_cycle(state)` [VERIFY: `grep -n "def advance_forward_cycle" user/scripts/lazy_core.py`] to the `if args.repeat_count:` block, mirroring the `--apply-pseudo` site at L6546-6552 [VERIFY: `grep -n "advance_forward_cycle" user/scripts/lazy-state.py`] which already calls it.
  - **Reconcile (the load-bearing detail — ⚖ KEEP-both decision from PHASES.md):** `advance_forward_cycle` becomes the authoritative forward-advance trigger on this probe path. `advance_run_counters` [VERIFY: `grep -n "def advance_run_counters" user/scripts/lazy_core.py`] must NOT also advance `forward_cycles` for the SAME real cycle. Two acceptable forms — pick the one that keeps the ISSUE-5 inflation regression green:
    1. Replace the `advance_run_counters(state)` call with `advance_forward_cycle(state)` on this path (the consume-gated advance is no longer the forward authority on the real-skill probe path; meta accounting via `--emit-dispatch`/`advance_meta_cycle` is untouched), OR

Document the chosen reconciliation in an inline comment naming which trigger owns the forward count on the probe path and why the other no longer double-counts.

## Blocking Issue Protocol

If a blocking issue is encountered at any point:

1. **Stop all in-progress work.** Do not launch new subagents.
2. **Commit and push any completed phases** not yet committed.
3. **Write `BLOCKED.md`** (canonical filename — see `~/.claude/skills/_components/sentinel-frontmatter.md`) in the bug dir, and print a blocking-issue report:

   ## Implementation Batch — Blocked
   **Completed phases:** [list with commit hashes]
   **Blocked phase:** Phase N: [title]
   **Reason:** [specific]
   **Recovery suggestion:** [what the user should do]
   **Remaining phases (not attempted):** [list]

4. **Do not work around the blocker.** The user (or orchestrator resolution mode) provides a resolution.

Blocking issues include: a `--test` baseline that cannot be reconciled after 2 attempts; a parity audit that stays red after both scripts are wired (indicates a real divergence to resolve); a quality-gate failure unfixable after 2 retries; any error requiring architectural decisions beyond these specs.

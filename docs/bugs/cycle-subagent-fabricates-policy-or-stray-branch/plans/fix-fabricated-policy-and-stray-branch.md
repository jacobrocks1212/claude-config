## Blocking Issue Protocol

If a blocking issue is encountered at any point during execution:

1. **Stop all in-progress work.** Do not launch new subagents.
2. **Commit and push any completed phases** not yet committed.
3. **Write `BLOCKED.md`** (canonical name — see `~/.claude/skills/_components/sentinel-frontmatter.md`) in the bug directory, and print a blocking-issue report:

   ## Implementation Batch — Blocked

   **Completed phases:** [list with commit hashes]
   **Blocked phase:** Phase N: [title]
   **Reason:** [specific description]
   **Recovery suggestion:** [what the user should do]

   **Remaining phases (not attempted):**
   - [list]

4. **Do not attempt to work around the blocker.**

Blocking issues include:
- A subagent failure that can't be fixed after 2 retry attempts
- A quality-gate failure that can't be fixed after 2 retry attempts (e.g. an `emit_cycle_prompt` residue refusal the prose edit introduced)
- A git push conflict that can't be resolved by rebase
- Any error that would require architectural decisions beyond the scope of the spec

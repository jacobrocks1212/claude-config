- **Implementation goal:** Add a new subsection (e.g. `## Full-probe-JSON read before routing (completeness, not just freshness)`) stating:
  - A routing/dispatch decision MUST be made against the **COMPLETE** current probe JSON. Never field-extract a subset of keys (no jq-style / pipe cherry-pick of e.g. `pending_hardening`, `terminal_reason`) and route on that subset.

  - This clause is **additive to** (does NOT replace) the existing atomicity (provenance) + freshness (same-turn) rules: atomicity governs WHERE the prompt came from, freshness governs WHEN, and this clause governs HOW COMPLETELY the probe output is consumed.
- **Spec requirements:** SPEC "Proven Findings" (contract does not require full-probe-JSON consumption; field-extraction permitted by omission). SPEC Open Question "Where to state it once" — prefer the shared component.

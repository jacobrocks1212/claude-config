### Phase: byref-dispatch-undercounts-forward-cycles P2 — Harden the consume watermark (defense-in-depth)

**Goal:** Make the consume watermark robust so a one-time downward census step (ring-cap eviction) or the meta `+1` over-absorb can never PERMANENTLY strand the gate. After Part 1 the forward count no longer depends on the census, so this is defense-in-depth for any remaining consumer of the watermark.

**Entry criteria:** Part 1 / Phase 1 complete (the forward-advance authority is `advance_forward_cycle`, so changing the consume-gate machinery here cannot regress the forward count).

**SPEC.md references:** `## Fix Scope` item 2 (Hardening); `## Evidence Collected` → `consumed_emission_count` docstring caveat, `advance_meta_cycle` `+1` over-absorb; Contributors A + B in `## Reproduction Steps`.

**WU-1 — Clamp the watermark against the non-monotonic census (lazy_core.py)**

- **Scope:** PHASES.md Phase 2 deliverables 1 + 2 + 3. Clamp `last_advance_consume_count` so it can never strand permanently; re-evaluate `advance_meta_cycle`'s unconditional `+1`; update the `consumed_emission_count` docstring caveat.
- **TDD:** yes
- **Files to create/modify:**
  - `user/scripts/lazy_core.py` — `advance_run_counters` at L7664 [VERIFY: `grep -n "def advance_run_counters" user/scripts/lazy_core.py`], `advance_meta_cycle` at L7746 [VERIFY: `grep -n "def advance_meta_cycle" user/scripts/lazy_core.py`], `consumed_emission_count` docstring at L7210 [VERIFY: `grep -n "def consumed_emission_count" user/scripts/lazy_core.py`].

- **Implementation goal:**
  - **Clamp:** when `advance_run_counters` and `advance_meta_cycle` write `last_advance_consume_count`, ensure the persisted watermark cannot exceed (or, on a census drop, permanently outrun) the live `consumed_emission_count()`. Concretely: a watermark write should not leave `last_advance_consume_count > consumed_emission_count()` in a way that strands future comparisons — e.g. clamp the written value to the live census, OR make the gate compare defensively so a census that has dropped below the persisted watermark re-arms (advances once) rather than no-oping forever. The ring cap is `_REGISTRY_RING_CAP = 64` [VERIFY: `grep -n "_REGISTRY_RING_CAP" user/scripts/lazy_core.py`] — eviction of consumed entries (`register_emission` while-loop) is the mechanism that drops the census below the watermark (Contributor B).
  - **`advance_meta_cycle` `+1`:** keep the absorb of the meta dispatch's own forthcoming consume (`last_advance_consume_count = consumed_emission_count() + 1`) but ensure it cannot ratchet the watermark permanently past the live count when meta dispatches outpace forward consumes (Contributor A). The clamp above should subsume this; if it does not, bound the `+1` so it never strands.
  - **Docstring:** add to `consumed_emission_count`'s docstring the caveat that the run-lifetime `last_advance_consume_count` watermark (which `advance_run_counters` compares against) is now clamped against a one-time downward census step, so eviction can no longer permanently strand the gate. The current docstring reasons only about the F2 double-probe debounce — extend it.
  - **CRITICAL — do NOT re-introduce the ISSUE-5 inflation:** the consume gate must still no-op a bare re-probe (no consume between two probes ⇒ no advance). The clamp must not turn a census drop into spurious repeated advances. WU-2's tests pin both directions.
- **Spec requirements:** Fix Scope item 2 — "make the watermark comparison robust to a non-monotonic oracle — e.g. clamp `last_advance_consume_count` to never exceed the live `consumed_emission_count()` it just observed … so `advance_meta_cycle`'s `+1` and ring-cap eviction can no longer permanently strand the gate."

- "Design constraints for the plan" — prefer the DLL directly under the Debug OutputPath over deeper config-variant copies (`bin\AutoTest\`, `bin\AutoTest-Firefox\`, …); preserve exit contracts 1/3/4; keep scoped to path resolution.
- Root-cause table (`Cognito.UnitTests` → `bin\Debug\` works; `Cognito.Forms.UnitTests` → `bin\` false-fires).

**Test expectations (the 4 cases — assert the RESOLVED PATH, and each must not throw):**
- (a) **`bin\Debug\<name>.dll` layout** → resolver returns that path. Build a temp `<ProjectDir>\bin\Debug\Foo.dll`; assert `Resolve-TestDllPath -ProjectDir <dir> -TestDll "Foo"` equals that full path.

Assert the returned path is `<ProjectDir>\bin\Foo.dll`. This case has no `Resolve-TestDllPath` to call against the pre-fix code → RED before the fix.
- (c) **Multi-copy layout** (`bin\Foo.dll` + `bin\AutoTest\Foo.dll`) → resolver returns the SHALLOW `bin\Foo.dll`, not the deeper `bin\AutoTest\Foo.dll`. Create both (make the deeper one *newer* via `Start-Sleep` before writing it, to prove shallow-wins beats newest-wins). Assert the returned path is the shallow `bin\Foo.dll`.
- (d) **Not-built** (no `Foo.dll` anywhere under `bin`, or no `bin` at all) → resolver returns the conventional `<ProjectDir>\bin\Debug\Foo.dll` path (which does not exist) and does NOT throw. Assert equality with that fallback path AND `{ Resolve-TestDllPath ... } | Should -Not -Throw`.

**Implementation goal (`Resolve-TestDllPath` behavior — satisfies all four cases + the SPEC constraints):**
1. Compute `$binDir = Join-Path $ProjectDir 'bin'`.
2. If `$binDir` does not exist → return the fallback `Join-Path (Join-Path (Join-Path $ProjectDir 'bin') 'Debug') "$TestDll.dll"` (case d). Never throw.
3. Otherwise `Get-ChildItem -Path $binDir -Recurse -Filter "$TestDll.dll" -ErrorAction SilentlyContinue`. If zero matches → same `bin\Debug\` fallback (case d).
4. **Selection rule (case c):** among matches, choose the one with the **fewest path segments below `bin\`** (shallowest). Compute depth per match as the number of directory separators in the path relative to `$binDir` (a match directly under `bin\` has depth 1; `bin\Debug\` and `bin\AutoTest\` have depth 2; `bin\AutoTest-Firefox\` depth 2; deeper variants higher). Sort ascending by depth, then **break ties by newest `LastWriteTime` descending**, and return the first match's `FullName`. This mirrors the artifact `dotnet test --no-build` loads for the default (Debug) configuration.
   - Note on cases (a) vs (b): when only `bin\Debug\Foo.dll` exists (a), it is the sole/shallowest match → returned. When only `bin\Foo.dll` exists (b), depth 1 < any Debug depth → returned. When both a shallow `bin\Foo.dll` and deeper copies exist (c), depth-1 wins regardless of mtime.
5. Wrap the whole body in `try { … } catch { return <bin\Debug\ fallback> }` so it never throws (case d hard requirement, mirrors `Test-StaleTestDll`'s fail-safe posture).
6. Guard the definition exactly like the existing helpers — it is a plain `function Resolve-TestDllPath([string]$ProjectDir, [string]$TestDll) { … }` placed **between** `Test-StaleTestDll` (ends line 82) and `Invoke-Main` (starts line 84). Being a plain function, it is dot-source-safe automatically; the file-level dot-source guard at the bottom [VERIFY: grep -n "MyInvocation.InvocationName" test-filtered.ps1 → line 169] already prevents `Invoke-Main` from running on import.

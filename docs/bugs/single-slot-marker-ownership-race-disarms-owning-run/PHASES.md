## Validated Assumptions

All load-bearing assumptions for this fix are **code-provable** (read directly from `lazy_core.py` / `lazy_guard.py` / `lazy-state.py` / `bug-state.py` during the Touchpoint Audit) ‚Äî no runtime-coupled assumption requires a live spike:

- **`--run-start` writes the marker bind-pending today** ‚Äî confirmed: `lazy-state.py::--run-start` (~7540) and `bug-state.py::--run-start` (~4611) both call `write_run_marker(pipeline=‚Ä¶, cloud=‚Ä¶, repo_root=‚Ä¶, max_cycles=‚Ä¶, attended=‚Ä¶)` with **NO `session_id` argument**, so `write_run_marker` (`lazy_core.py:7848`) always stamps `session_id: None` at run-start. This is the open bind-pending window that both Repro A and Repro B exploit.
- **The orchestrator's session_id IS available at `--run-start`** (SPEC Open Question 1, RESOLVED): the `--session-id` CLI arg already exists on BOTH scripts (`lazy-state.py:7061`, `bug-state.py:4475`) and is already consumed at `--marker-present` (`read_run_marker(session_id=args.session_id)`, `lazy-state.py:7349`) and `--marker-work-branch` (`marker_work_branch(session_id=args.session_id)`). The orchestrators have the session_id in-session at run-start. So a `--run-start`-time bind to the known owner session is structurally available ‚Äî it eliminates the bind-pending window at its source rather than racing to bind it later.
- **`read_run_marker` path B is the silent-disarm mechanism** ‚Äî confirmed at `lazy_core.py:7968‚Äì7981`: when the caller passes a non-None `session_id` AND the marker's `session_id` is non-None AND they differ, it returns `None` non-destructively. The owner reading `None` from its OWN session cannot today distinguish "no run live" from "my run live but stamped with a foreign session." That indistinguishability is what makes the disarm silent.
- **`bind_marker_session` is first-writer-wins + idempotent** ‚Äî confirmed at `lazy_core.py:8010‚Äì8051`: it reads the marker with NO session_id (path B disabled), stamps ONLY when `session_id is None`, and is a no-op once bound. So a wrong first bind wins permanently for the run; OVERWRITE is already impossible. The residual is purely the UNBOUND‚Üíwrong-bind transition.
- **`_bind_marker_on_allow` is the only normal-path binder post-WU-9.2** ‚Äî confirmed at `lazy_guard.py:377‚Äì401`: only an orchestrator ALLOW binds, and it calls `bind_marker_session` (itself idempotent). Once `--run-start` binds the owner, this becomes a confirming no-op (the owner's own session re-binding the already-correct slot) ‚Äî it does NOT need removal, only to stay idempotent.
- **The checkpoint-resume re-bind window is real** (Repro B) ‚Äî confirmed: the resume path re-calls `write_run_marker` (zeroing/`None`-ing `session_id`) then `restore_checkpoint_counters`, never re-stamping the prior owner (`lazy-state.py` ~7540‚Äì7577 / `bug-state.py` ~4611‚Äì4640). Binding session_id at `--run-start` closes this leg automatically ‚Äî the resume `--run-start` is itself a `--run-start` and will carry `--session-id`.
- **Shared helper ‚Üí both pipelines** ‚Äî `write_run_marker` / `read_run_marker` / `bind_marker_session` live in `lazy_core.py`; both state scripts import them. The `--run-start` bind-threading is a COUPLED-PAIR edit on `lazy-state.py` + `bug-state.py` (audited by `lazy_parity_audit.py::audit_state_script_parity`).
- **`lazy_coord.py`'s fencing pattern is the in-codebase precedent** (SPEC Proven Finding #3) ‚Äî `term_token` + `verify_fencing`. The owner-side DETECT path here is a lighter analog: the owner records its expected session at run-start (the bind), and the detect path distinguishes "no marker" from "marker present but foreign-stamped." No shared lease store ‚Äî reuse is by PATTERN.

**Design forks (SPEC ## Open Questions ‚Äî all scope-class, resolved IN-PLANNING per completeness-first D7; END-STATE behavior identical: a live run's guard stays armed for its owner and cannot be silently disarmed by a wrong bind):**

‚öñ policy: bind-unforgeability mechanism ‚Üí bind owner at `--run-start`
  Chosen: thread `--session-id` into the `--run-start` `write_run_marker` so the marker is born owner-bound, ELIMINATING the bind-pending window entirely ‚Äî the root of both Repro A and Repro B. The fencing-token alternative only DETECTS a wrong stamp after the window is exploited; closing the window at the source is the more complete fix and is structurally available (Open Question 1 = YES). Detection is still added (below) as the residual backstop for legacy/un-threaded run-starts.

‚öñ policy: detect/re-arm shape ‚Üí owner-recorded expected-session + re-arm CLI
  Chosen: the owner-recorded expected-session (the run-start bind IS that record) + a non-destructive detect helper distinguishing `absent`/`owned-by-me`/`foreign-stamped` + a `--reassert-owner` re-claim action. Lighter than a full `term_token` monotonic fence and reuses the existing `session_id` slot; the fencing-token model's extra machinery buys nothing once the bind window is closed. Same end-state: the owner can tell "no run" from "wrong-stamped run" and re-arm.

‚öñ policy: resume re-bind handling ‚Üí covered by the run-start bind
  Chosen: no separate resume-site code. Because the resume is itself a `--run-start` carrying `--session-id`, binding at run-start closes Repro B automatically ‚Äî one fix site, not two. A dedicated fixture proves the resumed run is owner-bound.

‚öñ policy: deterministic fixture design ‚Üí injected bind ordering
  Chosen: hermetic `test_lazy_core.py` fixtures that force a wrong-session `bind_marker_session` before the owner's allow (Repro A) and re-`--run-start` after a checkpoint (Repro B), using the existing `_set_state_dir` temp-dir + real `write_run_marker`/`bind_marker_session` producers ‚Äî no real timing race.

## Phase 1: Bind the marker to the owning session at `--run-start` (close the bind-pending window)

**Scope:** Eliminate the UNBOUND‚Üíwrong-bind window at its source. Thread `session_id=args.session_id` into the `--run-start` `write_run_marker(...)` call on BOTH state scripts so the run marker is born stamped with the orchestrator's known owning session ‚Äî never bind-pending. A foreign session can no longer be the first writer of the slot (it was never `None`), so `bind_marker_session`'s first-writer-wins protection now protects the CORRECT owner from the start. This closes Repro A (pre-allow bind race) AND Repro B (checkpoint-resume re-bind ‚Äî the resume `--run-start` carries `--session-id` too). `write_run_marker` already accepts `session_id` (no signature change); the edit is purely passing the already-available arg. Backward-compatible: a `--run-start` invoked WITHOUT `--session-id` (legacy/manual) still writes `session_id: None` and falls back to the unchanged `_bind_marker_on_allow` anchor ‚Äî no regression, the fix is additive.

## Phase 2: Owner-side detect + re-arm backstop (distinguish "no run" from "wrong-stamped run")

**Scope:** Give the true owner a way to (a) DETECT that its live marker is stamped with a foreign session ‚Äî distinct from "no marker at all" ‚Äî and (b) RE-ARM by re-claiming the slot, rather than silently fast-path-allowing forever. This is the residual backstop for the paths Phase 1 does not structurally close: a legacy/manual `--run-start` without `--session-id` (marker still bind-pending ‚Üí a foreign bind can still win) and any future code path that calls `bind_marker_session` outside the allow anchor. Add a non-destructive detect helper in `lazy_core.py` returning a three-way owner status (`absent` / `owned-by-me` / `foreign-stamped`) and an idempotent `reassert_marker_owner(session_id)` re-claim helper; expose the re-claim via an orchestrator-only `--reassert-owner` CLI action on both scripts, guarded like the other orchestrator-only ops.

**Deliverables:**
- [ ] In `lazy_core.py`, add `marker_owner_status(session_id, *, now=None) -> "absent" | "owned-by-me" | "foreign-stamped"`: NON-destructive (reuses `read_run_marker`'s age/corrupt rules for `absent`, but does NOT delete on a session mismatch ‚Äî it reports `foreign-stamped` instead of collapsing to `None`). `absent` = no live marker (or age-stale/corrupt). `owned-by-me` = live marker with `session_id` None (bind-pending ‚Äî the owner's, not yet stamped) OR equal to the caller. `foreign-stamped` = live marker whose non-None `session_id` differs from the caller. This is the detect path the SPEC's Proven Finding #4(b) requires ‚Äî it makes "no run" and "wrong-stamped run" DISTINGUISHABLE.
- [ ] In `lazy_core.py`, add `reassert_marker_owner(session_id, *, now=None) -> bool`: when `marker_owner_status(session_id)` is `foreign-stamped` on a live marker, atomically re-stamp the slot to `session_id` (`_atomic_write`) and return True (the re-arm / re-claim path ‚Äî Proven Finding #4(c)); for `absent` or `owned-by-me` it is a no-op returning False. The re-claim is the owner reclaiming its own run's guard ‚Äî only the run's actual orchestrator (which holds the run marker's `repo_root`-keyed state dir and its own session_id) can invoke it.
- [ ] In `lazy-state.py`, add an orchestrator-only `--reassert-owner` action (requires `--session-id`) calling `reassert_marker_owner(args.session_id)`; print the verdict JSON (`{reasserted: bool, prior_status: <status>}`). Gate it with `refuse_if_cycle_active("--reassert-owner")` FIRST (a cycle subagent is refused exit 3 with zero side effects ‚Äî the same orchestrator-only contract as `--run-start`/`--reorder-queue`).

- [ ] Docs: update `user/scripts/CLAUDE.md` ("Per-repo keyed state dir" / the owner-scoping + `bind_marker_session` prose) to document the `--run-start` owner bind (Phase 1), the `marker_owner_status` detect three-way + `reassert_marker_owner` re-arm, and that `_bind_marker_on_allow` is now a confirming no-op for the normal path (retained for legacy).

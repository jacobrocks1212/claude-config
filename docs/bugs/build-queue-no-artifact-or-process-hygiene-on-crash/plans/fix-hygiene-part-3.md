### Phase: P4 — Result-fidelity guard (zero-output / ambiguous-pass detection)

**Goal:** Close the third vector — `exit_code=0` trusted as a pass when the run produced no meaningful output. Two defensive layers: a zero-output guard in `test-filtered.ps1` and `result_fidelity` recording in the queue.
**Entry criteria:** Parts 1–2 complete — the runner awaits the build exit and writes the `hygiene` result sub-object.
**SPEC.md references:** Verified Symptom 5 (5-way-OR `/mstest` logged only the 239-byte header, `exit_code=0`, recorded as a pass); Theory 2 (CONFIRMED); Locked Decision 4 (two defensive layers land independent of the root-cause spike); Affected Area rows "Filtered test output" + "Result recording".

#### WU-1 — `test-filtered.ps1` zero-output guard + distinguished `exit 3`

- **Scope (PHASES P4 deliverable 2; deliverable 3 conditional-only per the spike note above):**
  - [ ] Track a `$resultLineCount` (incremented on each Passed/Failed test line matched) and a `$summarySeen` boolean (set when a summary line matches) inside the streaming `ForEach-Object` pipeline.
  - [ ] After the pipeline, capture `$dotnetExit = $LASTEXITCODE`.
  - [ ] If `$resultLineCount -eq 0 -and -not $summarySeen` ⇒ emit an explicit `⚠ No test results captured (zero tests matched filter or summary not parsed)` line and `exit 3` (the distinguished no-output code). Otherwise `exit $dotnetExit` (preserve the real verdict — a genuine all-pass still exits 0, a genuine failure still propagates its non-zero code).
  - [ ] **(Conditional, only if the operator has already recorded the spike finding as "unparsed-summary" in the bug dir):** widen the `:56` summary regex to also match the format the spike observed. If the spike is unrecorded or proved a filter-construction bug, SKIP this sub-item — the defensive guard ships alone and the conditional fix is an operator follow-up.
- **TDD:** no via automated harness (the script streams real `dotnet test` output; claude-config has no runtime to drive it). The counter logic is **observable** via the warning line and exit code during the PHASES Runtime Verification manual runs. Parse gate is the mechanical check.
- **Files to create/modify:**
  - `repos/cognito-forms/.claude/scripts/test-filtered.ps1` — add the counters, the post-pipeline exit capture, and the zero-output guard.
- **Test files:** none (no claude-config runtime; PHASES Runtime Verification covers it).
- **Test expectations:** N/A automated. PHASES Runtime Verification rows assert: a zero-match `/mstest` exits non-zero (3) with the warning; a genuine all-pass still exits 0.
- **Implementation goal:** Make an empty/zero-match test run announce itself instead of passing through `dotnet`'s ambiguous `0`. The script currently emits a line ONLY when it matches one of its regexes — passed (`^\s+Passed\s+\S`) [VERIFY: grep -n "Passed" repos/cognito-forms/.claude/scripts/test-filtered.ps1] (test-filtered.ps1:34), failed (`^\s+Failed\s+\S`) (`:39`), or summary (`Test Run (Passed|Failed)|^Total tests:|…`) [VERIFY: grep -n "Test Run" repos/cognito-forms/.claude/scripts/test-filtered.ps1] (test-filtered.ps1:56) — and tracks no "did I see any result line?" counter, so a zero-match run prints only the `Running tests…` header (`:18`) and passes through `dotnet`'s exit code. Increment `$resultLineCount` in the passed/failed branches; set `$summarySeen=$true` in the summary branch; declare both BEFORE the pipeline (the `$inFailBlock`/`$failBlockLines` vars at `:20-21` are the existing pattern to mirror). `$ErrorActionPreference` is already `Continue` here (`:10`) so `exit 3` is reachable. This is defensive layer 1 of Locked Decision 4 — it lands regardless of the root-cause spike.
- **Spec requirements:** Locked Decision 4 (zero-output guard in `test-filtered.ps1`); Verified Symptom 5; SPEC Proven Finding 5(a).
- **Batch:** 1 (solo — edits test-filtered.ps1).

This bug is a **two-part discriminator gap**: a self-inflicted environment transient (a zombie node process holding the AlgoBooth `:3333` sidecar named pipe after a `dev:restart`) is invisible to the harness's HTTP-only runtime-readiness gate AND has no escalation-immune terminal once it surfaces mid-cycle — so it is charged to the feature's `mcp-validation` validation-retry/escalation budget even though no code was wrong. Phase 1 (Leg A) enriches the orchestrator gate `lazy_core.ensure_runtime` with a sidecar-pipe (`is_connected`) readiness dimension; Phase 2 (Leg B) gives the runtime-up mcp-test cycle a runtime-readiness terminal so the in-cycle detection routes to `mcp-runtime-unready` (escalation-immune), not `mcp-validation`. The `validation_escalation` predicate is verified CORRECT and is NOT modified. Leg C (AlgoBooth `dev:kill` zombie-reaping) is deferred/out-of-repo.

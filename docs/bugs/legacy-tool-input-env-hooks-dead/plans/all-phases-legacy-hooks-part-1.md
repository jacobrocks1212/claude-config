### Phase: legacy-tool-input-env-hooks-dead P1 — Rewrite the two REGISTERED hooks on the stdin-JSON interface (+ pipe tests, matcher widen)

**Goal:** The push and terminal-kill protection hooks actually fire (deny-via-JSON on matching Bash *and* PowerShell payloads) instead of being a no-op on every input, proven by pipe tests.
**Entry criteria:** None (first phase).
**SPEC.md references:** `## Root Cause` (interface-contract-mismatch: hooks read `$TOOL_INPUT_command`, an env var the stdin-JSON interface never populates); `## Fix Scope` items 1–3, 5.

- **Test expectations (author each; confirm RED against the un-rewritten hooks):**
  - `block-terminal-kill.sh` **deny leg** — pipe `{"tool_name":"Bash","tool_input":{"command":"taskkill /F /IM node.exe"}}` → stdout parses to `hookSpecificOutput.permissionDecision == "deny"`, returncode 0. Repeat for `Stop-Process`, a bare `kill 1234`, `exit`, `wt.exe`.
  - `block-terminal-kill.sh` **allow legs** — `npx kill-port 3333` (the `kill-port` allowance) and a plain `ls` → returncode 0 with NO deny JSON.
  - `block-terminal-kill.sh` **malformed leg** — `not-json` on stdin → returncode 0, no deny.
  - `block-terminal-kill.sh` **PowerShell-payload deny** — `{"tool_name":"PowerShell","tool_input":{"command":"Stop-Process -Id 1234"}}` → deny (proves the tool-name-agnostic body).
  - `block-work-repo-git-push.sh` **deny leg** — in a throwaway git repo whose `user.email` is `jacob@cognitoforms.com`, pipe `{"tool_name":"Bash","tool_input":{"command":"git push origin main"}}` fired from that repo → deny. (Build the repo with an `_init_cognito_worktree`-style helper [VERIFY: `grep -n "_init_cognito_worktree" user/scripts/test_hooks.py`] — set `user.email jacob@cognitoforms.com`; the hook reads `git config user.email` from the payload cwd, so pass `cwd` in the payload.)
  - `block-work-repo-git-push.sh` **allow legs** — same push command but (a) prefixed `CLAUDE_PUSH_APPROVED=1` (bypass token) → allow; (b) in a repo whose `user.email` is a non-work email → allow; (c) a non-push command (`git commit -m x`) → allow.
  - `block-work-repo-git-push.sh` **malformed leg** — non-JSON stdin → returncode 0, no deny.
  - `block-work-repo-git-push.sh` **PowerShell-payload deny** — `{"tool_name":"PowerShell","tool_input":{"command":"git push origin main"}}` in a work-email repo → deny.
  - **Registration meta-tests** (mirror `test_longbuild_guard_registered_in_settings` [VERIFY: `grep -n "def test_longbuild_guard_registered_in_settings" user/scripts/test_hooks.py`]): parse `user/settings.json`, locate the PreToolUse block(s) registering `block-terminal-kill.sh` and `block-work-repo-git-push.sh`, and assert each hook's block matcher string contains BOTH `Bash` AND `PowerShell`. (These fail until WU-4 widens the matcher.)
- **Implementation goal:** Reuse `_run_bash(script, stdin_text, env)` [VERIFY: `grep -n "def _run_bash" user/scripts/test_hooks.py`] and `_base_env(state_dir)` [VERIFY: `grep -n "def _base_env" user/scripts/test_hooks.py`] for every pipe test; build payloads as `json.dumps({"tool_name":…, "tool_input":{"command":…}, "cwd":…})` mirroring the `_bqe_payload` shape [VERIFY: `grep -n "def _bqe_payload" user/scripts/test_hooks.py`]. Do NOT gate on `tool_name` (read the command regardless of which tool emitted it — this is what makes the PowerShell-payload deny test pass). Keep the deny messages informative (name the mobile-workflow reason, as the original stderr did).

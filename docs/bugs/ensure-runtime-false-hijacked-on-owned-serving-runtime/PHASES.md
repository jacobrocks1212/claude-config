## Validated Assumptions

Read directly from the named source sites — **code-provable**, no live runtime needed (the bug is a pure ownership-bookkeeping classification in deterministic Python, exercised by injected callables):

- **`_ensure_runtime_m4` `not owned` branch** (`lazy_core.py` ≈7518–7539): when `verify_runtime_ownership` is False, the handler reads `live_start = kernel_start_time_fn(lock['pid'])`. `live_start is None` ⇒ DEAD (recover); `live_start is not None` ⇒ **unconditional HIJACKED** strict fail-safe. **This is the false-positive site** — a live PID whose start_time matches the recorded lock start_time (only the *session* component of ownership diverges) is the self-booted-serving case and is currently mis-routed to terminal HIJACKED. (verified — Read 7518–7539)
- **`verify_runtime_ownership`** (`lazy_core.py` 8483–8514): returns True iff BOTH `controller_session_id == live_session_id` AND `recorded start_time == kernel start_time(pid)`. So a False return with a *matching* start_time means **the session component is the sole divergence** — exactly the distinguishing signal the soft-READY guard keys on. (verified — Read 8483–8514)
- **`_runtime_verdict`** (`lazy_core.py` 7330–7353): builds the verdict superset; `mcp_tools_present` is derived by `_mcp_tools_present_honest(payload, tool_name, health_code)` from the probe payload — already available at the decision site (the branch holds `code, payload`). A READY verdict with `ownership_verified=False` is representable today (the field is independent of `state`). (verified — Read 7330–7353)
- **`_hijacked_blocker`** (`lazy_core.py` 7356–7368): the genuine-foreign HIJACKED terminal_blocker text is unchanged by this fix — the foreign path (no matching lock PID via `live_start is None` ⇒ DEAD, OR a live PID whose start_time *diverges*) keeps returning terminal HIJACKED with this blocker. (verified — Read 7356–7368)
- **Coupling:** `--ensure-runtime` is **feature-pipeline-only** (`lazy-state.py` CLI + shared `lazy_core` helper); `bug-state.py` has NO `--ensure-runtime` handler and never calls `_ensure_runtime_m4`, so **no coupled-pair mirror is owed** (justified divergence — confirm green against `lazy_parity_audit.py`, per the SPEC Tests row + `user/scripts/CLAUDE.md`). (verified — `user/scripts/CLAUDE.md` `--ensure-runtime` COUPLING note)
- **Consumer routing** lives in `user/skills/lazy-batch/SKILL.md` Step 1d.0 (the sole consumer per `user/scripts/CLAUDE.md`; `/lazy-batch` is **user-level / AlgoBooth-flavored**, NOT under `repos/algobooth/` — path corrected from the SPEC's prose after recon) and, if it restates the gate, `user/skills/_components/lazy-batch-prompts/cycle-base-prompt.md`; it routes on the verdict's `state` (proceed-conjunction `state == READY AND health_code == 200 AND mcp_tools_present` — reads `state`/`health_code`/`mcp_tools_present`, NOT `ownership_verified`). A new non-terminal READY-with-`ownership_verified:false` proceeds to dispatch with no change to the terminal set — but the routing prose must explicitly recognize the soft-owned-unverified READY as "proceed", eliminating the per-cycle hand-reasoning workaround.

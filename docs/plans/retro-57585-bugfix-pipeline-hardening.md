## The generalized failure class (what we are hardening against)

> **A fix aimed at a *hypothesized* cause, certified "done" on *proxy-target* green.** The symptom→cause link was recorded as fact without tracing the symptom's actual production path (surface → … → data source); the fix suppressed the assumed value; every test asserted on that value (the fix's internal proxy) rather than on the symptom; the change shipped on green tests + a clean build; the symptom persisted because the assumed value was never on the symptom's path.

This is **not** overfit to "avatar chip / facet." It decomposes into two orthogonal, independently-generalizable seams, each of which would have stopped this failure and a class of siblings:

- **SEAM A — root-cause trace (front gate).** A "symptom X is caused by Y" finding may not lock unless the symptom's serving path is traced surface→source in code (or backed by cited runtime evidence for runtime-coupled claims), and the fix site lies **on** that path. *Catches: fixing the write path when the read path is broken; fixing a cache when the symptom is served fresh; editing a config the feature never consumes; suppressing a value the symptom never reads.*
- **SEAM B — symptom reproduction (back gate).** "Fixed" requires reproducing the *original* symptom and confirming it's gone at its reported surface — a green unit test on the fix's internal target is **not** sufficient. *Catches: green-tests-on-proxy while symptom persists; fix compiles but the wired path never executes; regression test pinning the internal value instead of the observable; any "unit-green = done" completion in a no-MCP repo.*

**Operator decisions locked for this plan:**
- **SEAM A enforcement: HARD BLOCK.** An `asserted` (untraced) causal link cannot conclude a bug SPEC or be planned. Interactive = refuse; `--batch` = `NEEDS_INPUT.md`.
- **SEAM B proof: SERVING-PATH REGRESSION TEST.** A red→green regression test that exercises the symptom's *actual serving path* (not the fix's internal proxy) satisfies the gate. A runtime/manual artifact (manual-testing doc or Selenium `local-ui-test`) is accepted as a *stronger* alternative but is not required.

### WU-1 — NEW component `root-cause-trace-gate.md` (SEAM A, hard block)

- **Action:** create
- **Repo path:** `claude-config/user/skills/_components/root-cause-trace-gate.md`
- **Purpose:** a docs-only gate that refuses to lock a bug's root cause unless the observed symptom's serving path is traced in code and the fix site sits on it.
- **Blast radius (injected by WU-3, WU-4):** `spec-bug`, `plan-bug`.
- **Draft — key rules the component must state:**
  1. **Scope:** runs whenever a bug SPEC records a causal finding of the shape *"observed symptom X is produced by value/code Y"* — before the SPEC may flip `Investigating → Concluded` (spec-bug) and before that finding may be planned (plan-bug).
  2. **Required artifact — the symptom serving-path trace.** For each `[VERIFIED]` symptom, record the chain from the **user-facing surface the symptom appears on** (UI component, API response field, log line, table row) back to the **data source that produces it**, each hop cited `file:line`. The chain must be produced by reading/greping the actual serving code, not inferred ("there is no other read-model, so it must be Y" is an inference, not a trace — explicitly forbidden as sufficient).
  3. **Fix-site-on-path rule.** The proposed fix must change a node **on the traced path** — i.e. the value/code being changed must be *read* on the symptom's path, not merely *related* to it. If the fix targets a value, the trace must show that value is consumed on the symptom's path.
  4. **`symptom-verified` ≠ `cause-traced`.** A reproduced/verified symptom does **not** imply a verified cause. Each causal finding carries an explicit label: `traced` (chain cited, fix-site-on-path shown) or `asserted` (hypothesis). A verified symptom may never be used to upgrade an `asserted` cause to fact.
  5. **Runtime-coupled claims** follow the `/investigate` four-attempt-trap: a runtime-coupled causal claim is **never** confirmed by reading source alone — it requires cited runtime/observed evidence. (Reference `investigate` + `systematic-debugging`; require their *artifact*, do not re-teach the method.)
  6. **HARD BLOCK.** An `asserted` link may not conclude a SPEC or be planned. Interactive → refuse with the specific untraced link named. `--batch` → write `NEEDS_INPUT.md` (`written_by: root-cause-trace-gate`, one decision naming the untraced symptom→cause link and the missing serving-path hops) per `_components/sentinel-frontmatter.md`.
- **Generalization guard (state in the component):** the gate keys on the generic symptom↔source relationship; it is surface-neutral (UI, API, CLI, log) and language-neutral.

### WU-2 — NEW component `symptom-reproduction-gate.md` (SEAM B, serving-path regression test)

- **Action:** create
- **Repo path:** `claude-config/user/skills/_components/symptom-reproduction-gate.md`
- **Purpose:** a docs-only completion gate: a bug cannot be declared Fixed without evidence the *original* symptom is reproduced-then-gone at its reported surface.
- **Blast radius (injected by WU-5, WU-6, WU-7, WU-8):** `completion-integrity-gate.md` (bug path), `fix`, `integration-verification.md`, `verification-before-completion.md`, `write-plan-cognito`.
- **Draft — key rules:**
  1. **Evidence ladder (strongest-first; the gate is satisfied by ≥ the required rung):**
     - **REQUIRED rung — serving-path regression test.** A red→green regression test that exercises the symptom's **actual serving path** (the path traced by WU-1's gate), verified **RED before the fix / GREEN after**. A test that asserts on the fix's *internal target* (a stored value, a facet, a private helper) does **NOT** satisfy this when that target is not itself on the symptom's serving path. For bug 57585 the qualifying test is one on `GetLinkedPersonAsync` / the `linked-person` endpoint — not on `CompositeEntryIndex.SubmitterPersonEntryId`.
     - **STRONGER (accepted, not required) — runtime/manual artifact.** A manual-testing doc (`/write-manual-testing-doc`) or a Selenium `local-ui-test` run observing the original symptom gone at the user surface.
  2. **Repro-recipe binding.** The test/artifact must map to the SPEC's `## Reproduction Steps` (the repro recipe captured at spec-bug time, WU-3). If the SPEC has no concrete repro recipe, that is itself a block.
  3. **Honesty ladder (mirror `_components/mcp/mcp-integration-test.md`):** "code changed + unit tests green on the internal target" is **NOT** verified — it is at most 50% PLAUSIBLE. Only a serving-path regression test (or stronger) reaches VERIFIED. "Tests green" and "symptom resolved" are distinct claims.
  4. **No `SKIP_MCP_TEST.md` bypass for bugs.** In a repo with no MCP surface, `SKIP_MCP_TEST.md` may satisfy the *MCP* validation-sentinel check but must **not** be read as satisfying symptom reproduction — the serving-path regression test is still required (it lives in ordinary unit-test land and needs no MCP).
- **Generalization guard:** surface-agnostic (backend/frontend/CLI/service); the "serving path" concept is defined relative to whatever surface the symptom was reported on.

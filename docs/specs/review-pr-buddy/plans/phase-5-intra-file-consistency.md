### Phase: review-pr-buddy P5 — Intra-File Reuse & Consistency Stage

**Goal:** Wire a dedicated intra-file reuse/consistency agent into the existing parallel Step 5b stage so both `review-pr` and `review-pr-buddy` flag (i) code that duplicates other code already in the same changed file and (ii) changes inconsistent with the file's surrounding conventions — across all substantively-modified substantive files, all triage tiers.

**Entry criteria:** Phases 1–4 complete (they are — all deliverables checked in PHASES.md). The `source:"reuse"` lane, `aggregate-findings.ts` reuse discovery, and the `cognito-consistency-checker` agent must exist (verified present in the Touchpoint Audit).

**SPEC.md references:** § "Reuse-candidacy stage (wired, parallel)" and § "Pipeline + script wiring" — this phase is the intra-file complement to the cross-file stage those sections describe; the host file's `main` version is the implicit baseline (no `baselines[]` seed).

#### Shared data contract (all WUs reference this — single source of truth)

- **New finding source:** `"intrafile"`. Rides the **investigation lane** in `post-process.ts` (fixed `effective_weight: 1.0`, never weight-filtered) — identical treatment to `"reuse"`.
- **Agent output file:** `{cacheDir}/agent-output/intrafile-{cluster-slug}.json`.
- **Findings schema:** the SAME shape the reuse agent emits (verified at `cognito-consistency-checker.md:229-273`): `{ group, findings: [{ file, line, severity, title, verdict, candidate, hypothesis, evidence{snippet,reference}, suggestion, blast_radius, negative_search_trail, escalation_candidate, specialist_domain }], escalations }`. After WU-1 this schema lives in the shared scaffold.
- **Verdict enum for intra-file:** duplication axis uses `reuse` / `refactor` (→ important); consistency axis uses `inconsistent` (→ nit) and `consistent` (→ dropped-with-negative-trail). `candidate` for a duplication finding is the in-file `file:line`/symbol it duplicates.
- **Verdict→severity (post-process):** `refactor`/`reuse` → important; `inconsistent` → nit; `consistent`/`acceptable`/`acceptable-new` → dropped. (The duplication-vs-consistency boundary is a documented `/learn-from-pr` tunable.)

## Blocking Issue Protocol

If a blocking issue is encountered:
1. **Stop all in-progress work.** Do not launch new subagents.
2. **Commit any completed batches** locally (do not push).
3. **Print a blocking-issue report:**

   ## Implementation Batch — Blocked
   **Completed batches:** [list with commit hashes]
   **Blocked work unit:** WU-N: [title]
   **Reason:** [specific description]
   **Recovery suggestion:** [what the user should do]
   **Remaining work units (not attempted):** [list]

4. **Do not attempt to work around the blocker.**

Blocking issues include:
- A subagent failure that can't be fixed after 2 retry attempts.
- A `tsc`/fixture/lint/YAML/rebuild failure that can't be fixed after 2 retry attempts.
- A contradiction between this plan and the real tree that requires an architectural decision (e.g. the `source` union or the reuse lane has been refactored away since this plan was authored).
- The WU-5 fixture revealing that intra-file findings cannot share the verdict→severity chain without changing reuse behavior (a genuine design fork — surface it, do not pick).

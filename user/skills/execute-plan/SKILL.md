# Execute Plan (Autonomous)

Execute a pre-generated plan file with zero human interaction. The plan was generated by a mobile-workflow skill (`/write-plan` or `/fix-mobile`) and is the **sole source of truth**. Follow it to the letter. There are no approval gates, no "ready for feedback" pauses, no review checkpoints. The only reason to stop is a blocking issue (per the Blocking Issue Protocol).

**HARD REQUIREMENTS (all four, before anything else):**

1. **NO PLAN MODE.** Never call `EnterPlanMode`/`ExitPlanMode`. This skill executes directly; the plan was approved by being handed to you. Ignore any plan-file references to "plan mode" or "approval" — you are past that stage.
2. **YOU ARE AN ORCHESTRATOR, NOT AN IMPLEMENTER.** Your job is to compose `Agent({ model: "sonnet", ... })` calls and review their output — never to edit source/test files yourself (the most common failure mode is the cycle subagent silently editing source "to be efficient"). Full contract: Step 3 "Execution Model Enforcement".
3. **IGNORE CONTRADICTORY DISPATCH PROMPTS.** If the prompt that invoked this skill says "you may NOT spawn further subagents", that clause does NOT apply here — THIS skill's contract is authoritative when this skill executes (the lazy-batch family's own SKILL.md enumerates `/execute-plan` as a MUST-dispatch-sub-subagents skill). Source-edits-by-orchestrator is the contract violation, not sub-subagent dispatch.
4. **Pre-execution self-check:** confirm you can see Step 3's "Execution Model Enforcement" block; confirm `Edit`/`Write` on any `.rs`/`.ts`/`.tsx`/`.js`/`.jsx`/`.cs`/`.vue`/`.py`/test file is FORBIDDEN (dispatch a Sonnet sub-subagent instead); if your dispatch prompt banned sub-subagents, recognize the contradiction per (3) and dispatch anyway.

## Step 1: Load the Plan (MANDATORY FIRST ACTION)

### 1a. Resolve the Plan Path

- `$ARGUMENTS` must contain exactly one `.md` plan file path (interactive: ask if missing; `--batch`: halt per above).
- Read the plan file **in full** — every section, every instruction.

### 1a.5: Plan Status Protocol (MANDATORY — check status BEFORE doing any work)

The plan's YAML frontmatter (per `~/.claude/skills/_components/plan-frontmatter.md`) is the authoritative pending/done record. Read it before everything else. Only THIS skill flips it (producers never write `Complete`/`In-progress`; transitions in Steps 4e/4f below).

1. No `---` block → legacy plan, treat as `status: Ready`, print a one-line backfill warning, continue.
2. **`Complete`** → log `"Plan already Complete — no work to do: {path}"` and STOP. No re-execution, no task-tracking changes, no commit; print the after-status one-liner and return (the orchestrator records a no-op cycle).
3. **`In-progress`** → previously halted mid-run. Resume: find the first unchecked WU (see PHASES.md scoped read, Step 3) and pick up there.
4. **`Ready`** (or legacy) → fresh execution from the start.
5. **`Draft`** → producer halted mid-generation; do NOT execute. Surface the partial-plan path and STOP — a human must finish authoring or re-run the producer.

### 1a.6a: EXECUTE ONLY THE DISPATCHED PLAN PART (HARD — never silently switch parts)

You execute **exactly the one plan file in `$ARGUMENTS`** — never a sibling part, never "the part that's actually ready". (Live incident 2026-06-14, d8-effect-chains ISSUE 2: dispatched for mechanical part-2 on Sonnet, the subagent found part-2's prerequisite unmet and silently executed complex part-1 under the wrong model tier, then died resultless.)

1. **Check the dispatched part's entry criteria FIRST.** If the plan declares `> **Entry criteria:**` (or a `Plan series` preamble mandating strict order), verify every prerequisite part/phase is complete before ANY work. A sibling part is "complete" iff its plan frontmatter is `status: Complete` — never infer from PHASES.md checkboxes alone.
2. **Unmet prerequisite → STOP.** Write `BLOCKED.md` with `blocker_kind: prerequisite-part-incomplete` naming the unmet prerequisite and this part's path, and return a clear blocked one-liner. Do NOT execute a different part to cope — in `--batch` this loud failure is what lets the orchestrator re-route.
3. **Never execute work from a different part.** "A different part should run first" is a routing decision — surface via BLOCKED.md.
4. **Model-tier integrity.** If the dispatched part's actual work exceeds its declared `complexity:` tier (e.g. complex cross-boundary work on a Sonnet/mechanical dispatch), STOP with `blocker_kind: model-tier-mismatch` rather than grinding under an under-powered model.

### 1a.6: CLOUD-SATURATION DECISION (mandatory before reporting "no work this cycle")

Before reporting a no-op cycle, perform this decision — skipping it is the documented cause of the `/lazy-batch[-cloud]` no-progress loop:

1. Enumerate this plan part's deliverable checkboxes: `python ~/.claude/scripts/phases-slice.py <PHASES.md> --phase <N> --checklist` for each phase in the plan's `phases:` frontmatter (path resolved from the plan's `feature_id` or cwd).
2. Read `<spec_path>/DEFERRED_NON_CLOUD.md` if it exists (its presence signals all cloud-runnable work is done).
3. For EVERY deliverable in this plan part: (a) checked off (`- [x]`)? OR (b) explicitly workstation-only with a corresponding `DEFERRED_NON_CLOUD.md` entry?

**YES to (a)/(b) for all** → flip the plan frontmatter to `status: Complete`, commit as `chore(<feature_id>): mark plan part N Complete (cloud-saturated)`, report the flip as this cycle's outcome, STOP. This is non-negotiable: `Complete` means **complete from the cloud-execution perspective** — the workstation tail is tracked by `DEFERRED_NON_CLOUD.md`, and the consumers of plan status (`lazy-state.py`, docs lint) interpret it exactly that way. Leaving the plan `In-progress` past saturation strands `/lazy-batch-cloud` in a no-op loop. The same flip applies in workstation mode when every unchecked in-scope deliverable is deferred-listed.

**NO** (any in-scope deliverable unchecked and not deferred) → real work exists; proceed to 1b.

> Orchestrator note: `lazy-state.py` performs this flip inline via `__flip_plan_complete_cloud_saturated__` before routing to Step 7a — this gate is the belt-and-suspenders backstop for callers that bypass the state script.

### 1b. Validate the Plan

Confirm the plan contains (or delegates via its Execution Policy pointer block): EXECUTION MODEL, MANDATORY RULES, Component Reference Card, Work Unit definitions, batch structure, Quality Gates step, commit step. **Pointer-only plans are the v2+ norm, not malformed** — skip warnings for any section the pointer block delegates to a contract file. If something is genuinely missing, print a warning but proceed — do NOT refuse to execute.

## Step 3: Execute the Plan

### The Golden Rule

**The plan file is gospel.** Do not: skip steps marked MANDATORY; reorder steps unless the plan allows it; substitute your judgment for the plan's instructions; inline-edit source/test files; skip quality gates, reviews, or PHASES.md updates; proceed past a blocking gate unsatisfied. **PHASES.md is your persistent memory** — after compaction it is the only record of what completed; treat its updates with the same gravity as committing code.

### Execution Contract (single source — READ THIS FIRST)

The autonomous-execution policy — EXECUTION MODEL, COMPONENT LOADING PROTOCOL, Component Reference Card, MANDATORY RULES, the full Execution Protocol (Phase-Selection Loop + per-batch Steps B.0–B.6 + Post-Phase Steps), Blocking Issue Protocol, and Completion report — is **single-sourced** in `~/.claude/skills/_components/execution-contract.md`. **`Read` it now and treat it as your operating contract** (once per run, not per batch).

**Repo-specialized contracts:** if the plan's pointer block names an ADDITIONAL contract (e.g. the Cognito lane contract `.claude/skills/write-plan-cognito/execution-contract-cognito-lanes.md`, repo-relative), `Read` that too, immediately after the generic one. Precedence, most-specific wins: the plan's "Plan-specific execution notes" → repo contract → generic contract.

The sections below are the **executor-specific** logic layered on top: plan-status protocol, part-integrity/saturation gates, scoped PHASES.md reads, per-WU checkbox discipline, ground-truth verification, compaction recovery, and the atomic gate+commit. Where the plan's own steps or a repo's `.claude/skill-config/` override a contract default, follow the plan/repo; otherwise the contract governs.

### PHASES.md scoped reads (SCRIPT-OWNED)

### Execution Model Enforcement

The contract's EXECUTION MODEL governs. Executor-specific reinforcement:

- **Fail-fast trip-wire (MANDATORY):** before ANY `Edit`/`Write`, ask "is this path a source/test file?" (`.rs`/`.ts`/`.tsx`/`.jsx`/`.js`/`.cs`/`.vue`/`.py`, anything under `tests/`, `src/`, `src-tauri/`). If yes — STOP; compose `Agent({ model: "sonnet", ... })` instead. The ONLY files you modify directly: `PHASES.md`, `IMPLEMENTATION_NOTES.md`, `CLAUDE.md`, the plan file (frontmatter status + WU checkboxes only), task tracking.
- **Sub-subagent dispatch contract (LOAD-BEARING):** every batch producing source/test changes MUST include ≥ 1 Sonnet test-agent dispatch (briefing: `~/.claude/skills/_components/tdd-test-agent.md`) and ≥ 1 Sonnet impl-agent dispatch (briefing: `~/.claude/skills/_components/implementation-agent.md`) — test-first, `agents_dispatched_per_batch ≥ 2`. (Lane plans override this via their contract's dispatch-census note: one inline-TDD lane agent counts as both.)
- **A batch completed with zero sub-subagent dispatches is a contract violation** — record `CONTRACT_VIOLATION: zero sub-subagents dispatched on batch <N>` in your final summary; never silently elide it.

### Per-Step Protocol

For each step/batch defined in the plan:

1. **Load components** — `Read` the component files listed for this step (per the contract's COMPONENT LOADING PROTOCOL: from disk, never from memory).
2. **Update task status** — mark the relevant task(s) `in_progress`.
3. **Execute (with pre-dispatch drift reconciliation)** — plan `file:line` anchors and named producers are **advisory** and may have drifted since authoring (especially off the plan's `source_branch`). Confirm each work unit's target (file, symbol, or call-site) still exists before dispatching; if absent, do NOT dispatch an agent to flail — locate the equivalent on the current branch, or record the WU as N/A/subsumed with a one-line rationale in PHASES.md, and continue. Then follow the plan's instructions exactly.
4. **Verify — Batch Review Gate (MANDATORY, BLOCKING).** When a batch's subagents complete, `Read` `~/.claude/skills/_components/subagent-review.md` from disk and run it IN FULL before any PHASES.md update, commit, or next batch. Load-bearing parts: (a) the Step 1.5 Ground-Truth Verification Gate — independently re-run every subagent's `GROUND-TRUTH OUTPUT` commands from your own shell and diff; a `no`/`mismatch` forces `NEEDS-REWORK` no matter how clean the prose report reads; (b) the TDD-discipline semantic-defect checks (vacuous/tautological tests hide behind green counts). A subagent's "N/N passing" is a hypothesis until re-verified. "Read a couple files, looks fine" is a protocol violation. Then run any additional gates the plan specifies.
5. **Tick the plan-part WU checkbox + update PHASES.md (BLOCKING GATE).**
   - **First tick this batch's `- [ ] WU-N` rows → `- [x]` in the PLAN PART's `## Work Units` checklist (machine source of truth — HARD).** `--verify-ledger --plan <plan_part>` reads `deliverables_done` from these rows; an unticked landed WU false-fails the completion gate and burns a recovery cycle. (A `- [ ] WU-N` under a `**Runtime Verification**`/`## MCP Integration Test` subsection stays unticked — the `/mcp-test` gate owns it.)
   - **Then update PHASES.md** per `~/.claude/skills/_components/phases-update.md`: phase **Status** + **Implementation Notes** (the durable human record) and best-effort deliverable ticks (documentation, not the gate). Re-read the phase slice to verify the write landed. Do NOT proceed until both are verified on disk.
6. **Update task status** — mark completed task(s) `completed`.
7. **Satisfy checklist** — must include "PHASES.md updated and verified for this batch".
8. **Commit the batch atomically** — make the commit the FINAL link of the SAME chained Bash command as the batch QG: `<batch QG> && git add -A && git commit -m "..." && git push` (see "Atomic gate+commit", Step 4, for rationale). A failing gate aborts before the commit.

### Blocking Issues

On a blocking issue (per the plan's Blocking Issue Protocol):

1. Stop all in-progress work.
2. **Flip the plan frontmatter to `status: In-progress`** (one-line edit), staged into the same commit as any partial work.
3. Commit and push completed work (including the flip).
4. **Remove the run marker:** `root=$(git rev-parse --show-toplevel 2>/dev/null || pwd) && rm -f ~/.claude/state/execute-plan/$(printf '%s' "$root" | md5sum | cut -c1-12).json`
5. Print the blocking-issue report the plan specifies.
6. **Do NOT work around the blocker.**

The same flip + marker removal applies on a `NEEDS_INPUT.md` halt. The next invocation reads `In-progress`, scans for the first unchecked WU, resumes, and rewrites the marker.

## Step 4: Completion

When all work units are complete (all tasks `completed`):

1. Follow the plan's **Completion** section (final QG run, completion report, etc.).
2. **Flip the plan frontmatter to `status: Complete`** — the authoritative skip signal for downstream tooling. Stage it into the final commit.
3. **Remove the run marker** (same command as Blocking Issues item 4).

> **Bug-fix plans (SEAM B) — the completion bar is symptom reproduction, not unit-green.** For a bug fix (`docs/bugs/**`, `kind: fix-plan`/bug `implementation-plan`), apply the "Bug fixed" rule in `~/.claude/skills/verification-before-completion/SKILL.md` + the evidence ladder in `~/.claude/skills/_components/symptom-reproduction-gate.md`: REQUIRED evidence is a serving-path regression test (red→green on the symptom's *actual serving path*, not the fix's internal target) or a runtime/manual repro. A batch closed on an internal-target unit test has NOT reproduced the symptom — treat the serving-path test as an unmet deliverable and route rework. (Distinct from the runtime/MCP carve-out below: a serving-path *unit* test needs no live runtime and IS yours to close.)

> **A runtime/MCP-validation gate is NOT unfinished plan work — flip to `Complete` anyway.** "All work units complete" means every WU this skill can ACT ON (code, tests, gates, commits). A live-runtime gate ("re-run `/mcp-test`", "validate on device", a `**Runtime Verification**` row) is owned by the pipeline's Step 9 — it cannot be closed without booting the runtime, which this skill does not do. Leaving the plan `In-progress` solely for such a gate routes `lazy-state.py` BACKWARD into re-dispatching this same plan, looping. Leave the verification row itself UNCHECKED in PHASES.md (`/mcp-test` ticks it); `remaining_unchecked_are_verification_only()` recognizes the remainder and advances. `In-progress` stays correct ONLY for a genuine BLOCKED/NEEDS_INPUT halt with implementation work pending. (State-machine backstop: a plan left `Ready`/`In-progress` with only verification rows open is routed to `__flip_plan_complete_stale__`, not a re-dispatch — still do the flip here so the backstop never fires.)

> **But the flip does NOT make the pending rows silent (MANDATORY — additive to the above).** Flipping `Complete` past unchecked runtime rows is correct ONLY when their pending state is ledgered and led with — the completion OUTPUT contract below is what keeps the flip honest in manual runs and no-MCP repos where `/mcp-test`/`__mark_complete__` never fire (the 57077 seam):

#### Atomic gate+commit — close the dual-ledger/commit gap (HARD REQUIREMENT)

The most expensive recurring failure: finishing the gates, then ending the turn in the gap between "gates passed" and "commit" (the harness auto-backgrounds the gate run and the completion notification reads as "done") — leaving the tree dirty and/or the dual ledger half-flipped, forcing a recovery dispatch. The structural fix: **the commit rides the SAME backgrounded job as the gates.**

1. **First, land the file Edits** that must precede the compound command: tick EVERY completed deliverable + verification checkbox in PHASES.md (exception — runtime-spike rows tick ONLY with a cited runtime artifact; otherwise leave unticked with a `NEEDS_RUNTIME:` note, per `phases-update.md`); flip the plan frontmatter to `Complete`; write Implementation Notes.
2. **Then ONE compound Bash command** chaining gates AND ledger commit AND push: `<gate1> && <gate2> && ... && git add -A && git commit -m "<message>" && git push`. `&&` gives both properties: a failing gate aborts before commit (never commit red); a passing run commits + pushes within the backgrounded job (ending your turn cannot strand the commit).
3. **Definition of done:** all verification boxes ticked, frontmatter `Complete`, tree clean, HEAD == origin. Final `git status --short` MUST print nothing; if it prints anything, stage/commit the remainder (re-running the gate chain) before reporting. Do NOT end your turn while the gate+commit job is still backgrounded — a mid-job completion notification is your cue to CONTINUE and verify, not stop. The same turn-end gate covers EVERY in-flight item at every seam, not just this completion job: an inner `Agent` dispatch whose returned report you have not consumed is in-flight work — dispatch-and-AWAIT; never end your turn "awaiting" a watcher/message/notification (inside a dispatched agent, ending your turn ends the agent — nothing re-invokes you). Canonical statement: `~/.claude/skills/_components/turn-end-gate.md`.

## Compaction Recovery Protocol

If you lose context to compaction:

1. **Do NOT panic or start over.** Task state persists. (The SessionStart(compact) hook `execute-plan-compact-reorient.sh` injects the active plan path from the Step 1d marker — use it if the conversation lost the path.)
2. `TaskList` immediately.
3. Re-read the plan file, then the execution contract(s) its pointer block names (generic + any repo contract).
4. The first non-completed task is your position.
5. **Verify the ledger for the last completed task:** `python ~/.claude/scripts/phases-slice.py <PHASES.md> --phase <id> --no-preamble` — confirm its deliverables are ticked and Implementation Notes written; if not, the update was lost — redo it now.
6. Re-read the Component Reference Card (contract/plan) and load the components for your current step.
7. Resume. **Never re-execute a completed task** — trust the task status.

## Plan File Output (Mobile Workflow)

This skill writes the plan to a Markdown file instead of entering plan mode. This enables a mobile workflow where plans are generated remotely and then executed by directing Claude Code to the plan file in a separate session.

**HARD REQUIREMENT — NO PLAN MODE:** Do NOT call `EnterPlanMode` or `ExitPlanMode` under any circumstances. Do NOT present the plan for interactive approval. The written plan file IS the deliverable of this skill invocation.

### Output Path Convention

Plans are colocated with the feature/bug documentation they target, inside a `plans/` subdirectory with informative names.

1. **If a PHASES.md was resolved:** write the plan to `{phases-dir}/plans/{informative-name}.md`
   - Example: `docs/features/audio/audio-vision/domains/d7-audio-analysis/plans/phase-3-sidecar-api.md`
   - Create the `plans/` directory if it doesn't exist: `mkdir -p {phases-dir}/plans`
2. **If a bug directory was resolved:** write the plan to `{bug-dir}/plans/{informative-name}.md`
   - Example: `docs/bugs/dc-bias-regression/plans/fix-pw-default.md`
3. **If standalone (no feature/bug directory):** write to `docs/plans/{informative-name}.md`
   - Example: `docs/plans/retro-audio-analysis.md`
   - Create `docs/plans/` if it doesn't exist

**Naming convention:** `{type}-{descriptive-slug}.md` where type is one of:
- `phase-N-{slug}` — single phase implementation (from /implement-phase)
- `all-phases-{slug}` — batch phase implementation (from /implement-phase-batch, /write-plan)
- `fix-{slug}` — bug fix (from /fix, /fix-mobile)
- `retro-{slug}` — retrospective improvements (from /retro)
- `spec-phases-{slug}` — spec decomposition (from /spec-phases-batch)
- `{custom-slug}` — anything else

### Write Protocol

1. **Create the plans/ directory** if it doesn't exist:
   ```bash
   mkdir -p {target-dir}/plans
   ```

2. **Write the plan** using the `Write` tool to the resolved path. The plan content is identical to what would have been presented in plan mode — fully self-contained, all sections included.

3. **Add a preamble** at the very top of the plan file:
   ```
   > **Mobile plan** — generated by `/{skill-name}` on {date}.
   > To execute: `/execute-plan {path}`
   > This plan is fully self-contained. The executing session needs no additional context.
   ```

4. **End with a copyable command** as the LAST thing output to the user. This must be a fenced code block so mobile users can tap to copy:

   ```
   Plan written to: {absolute-path}
   ```

   Then output this code block:

   ~~~
   ```
   /execute-plan {absolute-path}
   ```
   ~~~

   This is the final output. Do NOT add commentary after the code block.

### IMPORTANT: No Plan Mode

- Do NOT call `EnterPlanMode` or `ExitPlanMode`
- Do NOT present the plan for interactive approval
- The plan file IS the deliverable of this skill invocation

# Rebuild Agents

Regenerate agent prompts after rules have been updated in the YAML rules files.

**Plugin root:** `~/.claude/plugins/local-tools/plugins/cognito-pr-review`

## Source of Truth

Rules are stored as YAML files in `{plugin_root}/knowledge/rules/`:

| YAML File | Target |
|-----------|--------|
| csharp-architecture.yaml | rendered shard (sweep) |
| api-design.yaml | rendered shard (sweep) |
| frontend-vue.yaml | rendered shard (sweep) |
| performance.yaml | rendered shard (sweep) |
| testing.yaml | rendered shard (sweep) |
| code-consistency.yaml | rendered shard (sweep) + cognito-consistency-checker curated section |
| security.yaml | rendered shard (sweep) |
| template-binding.yaml | rendered shard (sweep) |

Sweep consumes the shards from the review cache (`{cacheDir}/rules/<category>.md`, copied in by `prep-pr.ts` from `{plugin_root}/knowledge/rendered/`).

Agent files are in `{plugin_root}/agents/`.

## Workflow

### 1. Read All YAML Rules

For each YAML file in `{plugin_root}/knowledge/rules/`:
- Parse the `rules:` list
- Note the `category` field
- Count rules per file

### 2. Map Rules to Agents

Using the table above, group rules by target agent:

- **cognito-consistency-checker**: code-consistency
- **sweep**: ALL categories (csharp-architecture + api-design + frontend-vue + performance + testing + code-consistency + security + template-binding)

Agents NOT updated by this process (they have different review approaches):
- **journey-planner** — produces journey files and validates triage, no static rules
- **triage** — classifies files into tiers based on PR context, no static rules
- **investigation** — deep-dive investigation with Solver-Verifier protocol, no static rules
- **cognito-intra-file-consistency** — per-cluster intra-file analysis via the shared reuse protocol, no static rules
- **synthesizer-v2** — narrative synthesis from post-processed findings, no static rules

### 3. Render Rule Shards (sweep's rule source)

The sweep agent does NOT embed rules — its rule corpus lives in per-category shard files that the prep script copies into each review cache (`{cacheDir}/rules/`). Regenerate the shards:

1. For each `{plugin_root}/knowledge/rules/<category>.yaml`, render `{plugin_root}/knowledge/rendered/<category>.md` (same basename as the YAML):
   - Header comment: `<!-- GENERATED shard — produced by /cognito-pr-review:rebuild-agents from knowledge/rules/<category>.yaml. Edit the YAML, then rebuild; do not edit this file. -->`
   - One H3 category heading (e.g. `### C# Architecture Rules`) with its `Category: \`<slug>\` | Multiplier: <category multiplier>` line
   - Convert each YAML rule into an H4 subsection: `#### <Title Case name> (\`<rule-id>\`)`, a `**Severity:**` line, the description, and anti_pattern/correct_pattern as fenced code blocks
   - Skip `source:` fields — rules are anonymous
   - **Do not embed numeric weights** — the live weights are the calibrated state file `~/.claude/state/cognito-pr-review/weights.yaml` (seeded from `knowledge/weights.yaml`); the prep script snapshots them into each cache as `weights-snapshot.json`
2. Verify coverage: every rule id from the YAMLs appears in exactly one shard; the union count equals the YAML total.
3. `agents/sweep.md` itself is only touched if a category was added/removed or the applicability mapping changed — update its "Rule Shards" manifest/applicability table (and the mirror list in `commands/review-pr.md` Step 5) to match.

### 4. Update Agent Files

For the curated rules section in **cognito-consistency-checker** (`## Code Consistency Rules (sub-capability — preserved)`):
1. Read the current agent file from `{plugin_root}/agents/cognito-consistency-checker.md`
2. Regenerate only rules that section already curates (it is a deliberate subset, not a full category embed)
3. **Preserve everything else**: frontmatter, intro text, review scope, cache instructions, output format

### 5. Re-embed Shared Components (checker agents)

The two reuse-class agents carry build-time embeds of the shared components (no runtime Reads):

| Component (source of truth) | Embedded into |
|-----------------------------|---------------|
| `~/.claude/skills/_components/reuse-discovery-protocol.md` | `agents/cognito-consistency-checker.md`, `agents/cognito-intra-file-consistency.md` |
| `~/.claude/skills/_components/pr-review-reuse-agent-scaffold.md` | `agents/cognito-consistency-checker.md`, `agents/cognito-intra-file-consistency.md` |

For each agent file, locate every `<!-- COMPONENT_START: <name> -->` / `<!-- COMPONENT_END: <name> -->` marker pair and replace the content between the markers with the current component text, preceded by the generated banner comment (`<!-- GENERATED — embedded from ~/.claude/skills/_components/<name> by /cognito-pr-review:rebuild-agents. … -->`). The component files remain the single source of truth — never edit the embedded copies inline.

### 6. Report Changes

After updating, report:

```markdown
## Agent Rebuild Summary

**Rules processed**: X total rules across Y categories

**Shards regenerated**: knowledge/rendered/*.md (8 files, [total] rules; coverage verified)

**Agents updated**:
- cognito-consistency-checker: [count] curated rules + component re-embeds
- cognito-intra-file-consistency: component re-embeds
- sweep: [unchanged | shard manifest updated]

**Agents unchanged**: synthesizer-v2, journey-planner, triage, investigation

**Rebuild complete!**
```

## Notes

- Always preserve agent frontmatter (name, description, model, color)
- Preserve cache-based file access instructions and scope boundary sections
- Preserve output format sections
- Don't remove agent-specific guidance (review philosophy, context notes)
- Convert YAML `anti_pattern`/`correct_pattern` into fenced code blocks in markdown
- If a YAML file has new rules not previously in the agent, add them under the appropriate heading
- Never embed `source:` fields from YAML rules into agent prompts — rules are anonymous

### Phase: weight-calibration-feedback-loop P4 — Close the feedback loop (disposition-driven recalibration)

**Goal:** Feed operator dispositions into the EMA. Buddy recalibrates immediately from dispositions (dismiss→signal `0`, kept→`1`, per recovered `rule_id`/`source`); non-buddy `review-pr` writes a `pending-calibration` marker for a later `/learn-from-pr`. Recalibration runs silently with a printed summary — no opt-in prompt.

**Entry criteria:** Phase 1 (`source_weights` + `processed-findings.json` join keys exist) and Phase 3 (recalibrated weights take effect without rebuild) Complete.

**SPEC.md references:** Proposed Fix **R2** (auto-recalibrate on completion, asymmetric signal); Evidence "Calibration exists but reads the wrong signal" (`learn-from-pr.md §2.5.2/§2.5.4`); Reuse Ledger "EMA weight update — Extend" + "Disposition capture — Reuse as-is".

**Disposition→weight join contract (binds WU-4a and WU-4b — derived from the audit, this is the crux of Phase 4):**
- A `buddy-session.json` disposition record carries `{ finding_ref, source, severity, note? }` (no `rule_id`). `finding_ref` is `"<file:line or id>"`.
- `processed-findings.json` findings carry `source`, `file`, `line`, and — **for sweep findings only** — `rule_id`. Investigation/intrafile/reuse findings carry NO `rule_id`.
- **Recovery join:** match each disposition to its processed finding by `(file:line == finding_ref, source)`. From the matched finding: sweep → recover `rule_id` → update `rule_weights[rule_id]`; non-sweep → recover `source` → update `source_weights[source]`.
- **Signal:** `dismiss` → `0.0`; any kept severity (`blocking`/`important`/`suggestion`) → `1.0`.
- **EMA:** reuse the existing formula `new = α·signal + (1-α)·old`, `α = ema_alpha` (`weights.yaml:4`, `0.25`). Increment `data_points`.
- **No fabricated signal:** a session with zero dispositions moves nothing. Undispositioned/abandoned findings yield no signal (correct).
